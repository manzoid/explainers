An array problem, in memory

Here is one small interview problem: reverse an array in place. We will not just solve it. We will watch it run in memory, cell by cell, and name every change in the vocabulary this series has been building. By the end, tracing code by hand should feel like a skill you own, not a trick.

This is the last page in the series, and it puts the earlier ones to work. State gave us the idea that a running program is information changing step by step. Memory gave us the cells that information lives in. Arrays gave us a row of cells reached by an index. And solving a single-function problem gave us a way to approach a question like this one. Now we take a real problem and run it in our heads, the way the machine would.

1The problem

Given an array of numbers, reverse it: the first item ends up last, the last ends up first, and so on toward the middle. Do it in place, meaning change the array you were given instead of building a new one alongside it. In place is the harder and more interesting version, because it forces you to think about the exact cells.

The clean way to do this uses two pointers. Here "pointer" just means an index variable, a number that says which cell we are looking at. One index, left, starts at the first cell. The other, right, starts at the last. We swap the two items they point to, then step both indices one cell inward, and repeat until they meet in the middle.

def reverse(a):
    left = 0
    right = len(a) - 1
    while left < right:
        a[left], a[right] = a[right], a[left]  # swap
        left = left + 1
        right = right - 1
    return a

2What is in memory before we start

Say the array is [5, 8, 2, 9]. From the arrays page, that is four cells in a row, reached by index 0 to 3. Alongside them sit two more small pieces of state, the index variables left and right, each holding a single number.

So the whole state we care about is: four array cells, plus left and right. That is the logical state from the state page, the part we actually reason about. We set left to 0 and right to 3, and now every step of the run is going to overwrite some of these cells. Each overwrite is a mutation: the old value is gone, the cell holds the new one.

3The film, snapshot by snapshot

The state page said a run is a film: take a snapshot at every step and play them in order. Here is the whole loop as three snapshots. Read down the rows, and watch the array cells and the two pointers change together.

4Narrate one step in the vocabulary

Tracing is nothing more than saying out loud what changed at each step, in the plain words of the series. Do the first step carefully, then the diagram carries the rest.

Step 1. left is 0, right is 3, and 0 < 3 is true, so we run the body. The swap reads cell 0 (which is 5) and cell 3 (which is 9), then overwrites cell 0 with 9 and cell 3 with 5. That is two mutations to the array. Then left becomes 1 and right becomes 2: two more mutations, to the index variables this time. The array is now [9, 8, 2, 5].

That is the whole rhythm of the loop: two mutations to swap, two to step the pointers inward. Steps 2 and 3 in the diagram are the same rhythm again, until the pointers cross and the loop stops. Say them to yourself and check them against the rows.

5Why the swap needs care

Look closely at one thing, because it is where beginners write a bug. To swap two cells you cannot just do a[left] = a[right] and then a[right] = a[left]. The first line overwrites cell left, and the old value that lived there is now gone. (On the state page, gone means gone: an overwrite destroys what was there.) The second line then copies the wrong number back, and both cells end up holding the same value.

The fix is to hold the old value somewhere before you overwrite it: save a[left] in a temporary variable, copy a[right] into a[left], then copy the saved value into a[right]. That temporary is one more small piece of state, one more cell. The Python line a[left], a[right] = a[right], a[left] does this saving for you behind the scenes, but the machine still needs the extra hold. Knowing that is knowing the state, not just the syntax.

Almost every hard bug is a wrong belief about the state. Trace the cells by hand and any wrong belief becomes visible right away.

6Check the edges by tracing them too

The single-function page pushed you to test the awkward inputs, not just the tidy one. Trace them the same way, in memory, and they answer themselves.

An empty array: left is 0, right is -1, and 0 < -1 is false, so the loop never runs. Correct, an empty array reversed is still empty. A single item: left and right both start at 0, 0 < 0 is false, loop never runs, and one item reversed is itself. An odd length, like five items: the pointers meet exactly on the middle cell, which does not need to move, and left < right becomes false right there. The < in the condition, not <=, is what makes the middle cell sit still. You do not have to guess whether these work. You trace them.

7What you actually built

Interview problems on arrays, strings, and lists are, underneath, exactly what we just did: a handful of cells, a few index or pointer variables, and a loop that mutates the state forward until it stands for the answer. When you can read a problem and see the cells, name the mutations, and step the film in your head, you can both solve those problems and explain your solution out loud while you do it. That is the payoff of treating this abstract series as one concrete skill.

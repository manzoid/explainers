An array problem, in memory

You can read this code, follow every line, and still not be sure what it does to the array. That gap is exactly what interviewers probe, and closing it is a different skill from reading code. This page closes it for one real problem: reverse an array in place.

We will name every change with three plain words. State is everything the program is holding right now. A mutation is overwriting a value in place, so the old one is gone. A snapshot is the whole state frozen at one step. (These come from the earlier pages: a running program is state changing step by step, that state lives in memory cells, and an array is a row of cells you reach by an index.) Here we take one problem and run it in your head the way the machine would, so those words become something you can do.

1The problem

Given an array of numbers, reverse it: the first item ends up last, the last ends up first, and so on toward the middle. Do it in place, meaning change the array you were given instead of building a new one alongside it.

Why insist on in place? Because it costs no extra memory, and that matters in two places you care about. In an interview, it is how the interviewer checks that you understand extra memory is not free. In production, copying a large array can use up the memory the machine has, so the in-place version is the one that keeps working when the data gets big.

The clean way to do this uses two index variables, one starting at each end. left starts at the first cell, right starts at the last. We swap the two items they mark, then step both indices one cell inward, and repeat until they meet in the middle. (This is the standard interview pattern called two pointers. Here the word "pointer" means an index variable, a number that says which cell we are looking at, not a memory address as on the pointer page.)

def reverse(a):
    left = 0
    right = len(a) - 1
    while left < right:
        a[left], a[right] = a[right], a[left]  # swap
        left = left + 1
        right = right - 1
    return a

2What is in memory before we start

Say the array is [5, 8, 2, 9]. From the arrays page, that is four cells in a row, reached by index 0 to 3. Alongside them sit two more small pieces of state, the index variables left and right, each holding a single number.

So the whole state we care about is: four array cells, plus left and right. That is the logical state from the state page (the part relevant to the algorithm, not every byte in memory). We set left to 0 and right to 3, and now every step of the run overwrites some of these cells. Each overwrite is a mutation: the old value is gone, the cell holds the new one.

3The film, snapshot by snapshot

The state page said a run is a film: take a snapshot at every step and play them in order. Here is the whole loop as three snapshots. Read down the rows, and watch the array cells and the two indices change together. One thing to expect: in Step 3 the left and right labels appear on opposite sides, because by then the indices have passed each other. That is the signal to stop.

4Narrate the steps in the vocabulary

Tracing is nothing more than saying out loud what changed at each step, in the plain words of the series. Here are the first two steps, said slowly.

Step 1. left is 0, right is 3, and 0 < 3 is true, so we run the body. The swap reads cell 0 (which is 5) and cell 3 (which is 9), then overwrites cell 0 with 9 and cell 3 with 5. That is two mutations to the array. Then left becomes 1 and right becomes 2: two more mutations, to the index variables this time. The array is now [9, 8, 2, 5].

Step 2. left is 1, right is 2, and 1 < 2 is true, so we run the body again. The swap reads cell 1 (which is 8) and cell 2 (which is 2), then overwrites cell 1 with 2 and cell 2 with 8. Two mutations. Then left becomes 2 and right becomes 1. The array is now [9, 2, 8, 5].

The rhythm repeats: two mutations to swap, two to step the indices inward. In Step 3 the check is 2 < 1, which is false, so the loop stops. Say these to yourself and check them against the rows.

5Why the swap needs care

Look closely at one thing, because it is where beginners write a bug. It is tempting to swap two cells like this:

a[left] = a[right]   # WRONG
a[right] = a[left]

The first line overwrites cell left, and the old value that lived there is now gone. (On the state page, gone means gone: an overwrite destroys what was there.) The second line then copies the wrong number back, so both cells end up holding what was in a[right]. The original left value is lost.

The fix is to hold the old value somewhere before you overwrite it, using one more cell, a temporary variable:

temp = a[left]       # save the old left value
a[left] = a[right]
a[right] = temp      # put the saved value back

That temp is one more small piece of state. The Python line a[left], a[right] = a[right], a[left] does this saving for you behind the scenes, but the language still has to hold the old value somewhere before it can overwrite. JavaScript can hide the temporary the same way, with the same shape:

[a[left], a[right]] = [a[right], a[left]]   // JavaScript

The temporary variable is still there, just out of sight. Knowing what is really happening is knowing the state, not just the syntax.

Almost every hard bug is a wrong belief about the state. Trace the cells by hand and any wrong belief becomes visible right away.

6Check the edges by tracing them too

The single-function page pushed you to test the awkward inputs, not just the tidy one. Trace them the same way, in memory, and they answer themselves.

An empty array: left is 0, right is -1, and 0 < -1 is false, so the loop never runs. Correct, an empty array reversed is still empty. A single item: left and right both start at 0, 0 < 0 is false, loop never runs, and one item reversed is itself.

The odd length case is the one that shows why the stopping condition is left < right and not left <= right. Take [5, 8, 2, 9, 1]. After two swaps, left is 2 and right is 2: both landed on the same middle cell. With left < right, the check is 2 < 2, which is false, so the loop stops and never touches cell 2. That is correct, because the middle item of a reversed array stays exactly where it was. Had we written left <= right, the loop would swap that cell with itself, harmless here but the wrong habit. You do not have to guess whether these work. You trace them.

Your turn. Take [7, 4, 1, 9, 3] and trace it in your head: write down left, right, and the five cells after each step, and stop when the check goes false. You should reach [3, 9, 1, 4, 7], with cell 2 never touched.

7The same shape, over and over

Most interview problems on arrays, strings, and lists have the shape you just traced: a handful of cells, a few index variables, and a loop that mutates the state forward until it stands for the answer. The harder-looking ones only change where the indices start and when they move.

Take removing duplicates from a sorted array in place. Same array, but now the two indices do different jobs instead of walking toward each other. One, call it write, marks the end of the kept part at the front. The other, read, scans ahead looking for the next new value. When read lands on a value that is new, we copy it just past write and step write forward. Here are two snapshots of that on [1, 1, 2, 3, 3].

Partitioning an array, merging two sorted arrays, the sliding window, prefix sums: the arrays page lays out this family. Each is the same picture with a different rule for where the indices start and when they move. Trace any of them the way you traced this one.

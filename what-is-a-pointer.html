What a pointer is: indirection

A pointer is a value whose content is an address: a box that holds a location, not a thing. It says "the thing is over there," and following it is one hop to the real value. That extra hop has a name: indirection.

The bytes and addresses page showed that memory is a long row of numbered cells, each with an address. The variables page showed a name standing for a value at one of those addresses. A pointer is the next step: a value that is an address.

You may write JavaScript or Python and never type a pointer. You lean on them constantly anyway. Whenever a variable holds an object or an array, it holds a pointer to that object, not the object itself. That is why this can bite you:

let a = { x: 1 };
let b = a;      // b now points at the same object as a
b.x = 99;       // a.x is 99 too, though you never touched a

A shaky picture of pointers is what makes that surprise, and a whole family of bugs like it, hard to see coming. This page draws the picture so they stop.

1A pointer is just a number, read as a place

A normal variable holds a value you want directly: the cell for count holds 7, and 7 is the answer. A pointer's cell holds a number too, say 4000, but you do not want 4000. You want whatever lives at address 4000.

Nothing in the bits marks a pointer as special. An address is just a number, and the cell holding it looks like any other cell. What makes it a pointer is how you read it: not "the value I have" but "where the value is." As the state page put it, bits mean nothing until something reads them a certain way.

2Following it is one hop

To go from the pointer to the real value, you follow the address it holds. That step has a name: dereferencing. You read the pointer, get a location, and go read what is at that location. One hop.

Some languages make you write both steps by hand. In Go, the arrow above looks like this:

count := 7
p := &count       // p holds the address of count
fmt.Println(*p)    // *p follows the address: prints 7

JavaScript, Python, Ruby, and Java do the same hop, but hidden. You never type the & or the *; when you write user.name, the language quietly follows the reference to the object, then reads the field. The arrow is there either way, on the screen or under it.

3Why go through an address at all

Holding a location instead of the thing sounds like extra work. It buys three things a plain copy cannot.

You can let two places share one thing. Two pointers holding the same address both reach the same value, so a change through one shows up through the other. One thing, two ways in.

You can pass something big without copying it. Hand a function a 10,000-item array by value and you copy all 10,000 entries; hand it the address and you copy one small number. The function reaches the same array through that address.

You can hold a thing that grows at runtime. Keep appending to a list and it gets bigger, but the pointer to it stays the same small, fixed size. You hold the address; the thing at the other end is free to change size.

That same flexibility is where bugs come in. Because the pointer is itself just a value in a cell, you can overwrite it, and two acts that look alike are not:

let a = { x: 1 };
let b = a;        // a and b point at one object
b.x = 99;         // change the thing:  a.x is now 99 too
b = { x: 0 };     // repoint b:  a is untouched, still { x: 99 }

The line b.x = 99 reaches through the arrow and edits the shared object. The line b = { x: 0 } only bends b's arrow at a new object; nothing at the old address moved. Repointing the arrow is not the same as changing what is at the end of it, and confusing the two is a classic bug. This is also the whole story behind what your language calls reference types: b = a copies the address, not the object.

4Copy vs share, and aliasing

Once a value can be an address, two questions run under everything: does this code hold the thing, or a pointer to it? And if it is a pointer, who else holds the same address?

That is the difference between copy and share. Copy gives each side its own thing; share hands the same address to more than one holder. When two names reach one thing, they are aliases, and a change through one silently changes what the other sees. This bites hardest across a function call, where you feel safe:

function markDone(todo) {
  todo.done = true;   // edits the caller's object, not a copy
}
markDone(myTodo);     // myTodo.done is now true

You passed myTodo in and the function changed it, because the argument was a pointer to your object, not a fresh copy of it. That is an aliasing bug: the b.x = 99 surprise from the top of the page, grown up.

A pointer is indirection: one extra hop between a name and its value. Every copy-vs-share decision, and every aliasing bug, is really a question about that hop and who else is standing at the other end of it.

One last edge: the arrow only helps if it points at a real value. A pointer can hold a special address meaning "points at nothing" (a null pointer), or an address that used to be valid but no longer is (a dangling pointer). Follow either and the hop lands in the wrong place, or crashes. Those sharp edges get their own page: null and dangling pointers. Keep the arrow picture in mind and they stop looking like magic.

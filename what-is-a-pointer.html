What a pointer is: indirection

A pointer is a value whose content is an address: a box that holds a location, not a thing. It says "the thing is over there," and following it is one hop to the real value.

The bytes and addresses page showed that memory is a long row of numbered cells, each with an address. The variables page showed a name standing for a value at one of those addresses. A pointer is the next step: a value that is an address.

You may write JavaScript or Python and never type a pointer. You lean on them constantly anyway. Whenever a variable holds an object or an array, it holds a pointer to that object, not the object itself. That is why this can bite you:

let a = { x: 1 };
let b = a;      // b now points at the same object as a
b.x = 99;       // a.x is 99 too, though you never touched a

A shaky picture of pointers is what makes that surprise, and a whole family of bugs like it, feel like magic. This page draws the picture so they stop.

1A pointer is just a number, read as a place

A normal variable holds a value you want directly: the cell for count holds 7, and 7 is the answer. A pointer's cell holds a number too, say 4000, but you do not want 4000. You want whatever lives at address 4000.

Nothing in the bits marks a pointer as special. An address is just a number, and the cell holding it looks like any other cell. What makes it a pointer is how you read it: not "the value I have" but "where the value is." As the state page put it, bits mean nothing until something reads them a certain way.

2Following it is one hop

To go from the pointer to the real value, you follow the address it holds. That step has a name: dereferencing. You read the pointer, get a location, and go read what is at that location. One hop.

Drawn this way, a pointer is just an arrow from one cell to another. That picture is the whole idea.

3Why go through an address at all

Holding a location instead of the thing sounds like extra work. It buys three things a plain copy cannot.

You can let two places share one thing. Two pointers holding the same address both reach the same value, so a change through one shows up through the other. One thing, two ways in.

You can refer to something big without copying it. Passing an address is cheap; passing the thing means copying every byte. A single small address can stand in for a huge table.

You can handle a thing whose size you did not know up front. A pointer stays small and fixed no matter how big the thing at the other end is.

And because the pointer is itself just a value in a cell, you can overwrite it. Here is where pointer code most often goes wrong, because two acts that look alike are not:

let a = { x: 1 };
let b = a;        // a and b point at one object
b.x = 99;         // change the thing:  a.x is now 99 too
b = { x: 0 };     // repoint b:  a is untouched, still { x: 99 }

The line b.x = 99 reaches through the arrow and edits the shared object. The line b = { x: 0 } only bends b's arrow at a new object; nothing at the old address moved. Repointing the arrow is not the same as changing what is at the end of it, and confusing the two is a classic bug.

4You are already using pointers

If you write C or Go, you type pointers on purpose. If you write JavaScript, Python, Ruby, or Java, you rarely see the word, yet every variable that holds an object holds a pointer to it. The language dereferences for you: user.name quietly does the one hop from the reference to the object, then reads a field.

This is why assigning one object variable to another does not copy the object. b = a copies the address, so both now aim at one object. That is the whole story behind what your language calls reference types. Whether the arrow is on the screen or hidden, it is there.

5Why this matters: copy vs share, and aliasing

Once a value can be an address, two questions run under everything: does this code hold the thing, or a pointer to it? And if it is a pointer, who else holds the same address?

That is exactly the difference between copy and share. Copy gives each side its own thing; share hands the same address to more than one holder. When two names reach one thing, they are aliases, and a change through one silently changes what the other sees. Aliasing bugs are what happens when you forgot that, the b.x = 99 surprise from the top of the page, grown up.

A pointer is indirection: one extra hop between a name and its value. Every copy-vs-share decision, and every aliasing bug, is really a question about that hop and who else is standing at the other end of it.

6When the arrow leads nowhere

An arrow only helps if it points at a real value. A pointer can hold a special address that means "points at nothing" (a null pointer), or an address that used to be valid but no longer is (a dangling pointer). Follow either one and the hop lands in the wrong place, or crashes.

These are the sharp edges of the same simple idea, and they get their own page: null and dangling pointers. Keep the arrow picture from this page in mind and those failures stop looking like magic. A pointer is a box holding a location; a bad pointer is a box holding a location you should not follow.

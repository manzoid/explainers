What a pointer is: indirection

A pointer is a value whose content is an address: a box that holds a location, not a thing. It says "the thing is over there," and following it is one hop to the real value. That extra hop has a name: indirection.

You may write Python or JavaScript and never type a pointer. You lean on them in every line anyway. This bug is why it matters:

let a = { x: 1 };
let b = a;      // b now points at the same object as a
b.x = 99;       // a.x is 99 too, though you never touched a

You changed b and a moved. That surprise, and a whole family of bugs like it, comes from one idea: when a variable holds an object or an array, it does not hold the object. It holds a pointer to it. An address is just a number that names one slot in memory, the way a house number names one house on a street. A pointer is a value that is such a number. This page draws that picture so the bugs stop surprising you.

If you want the two steps this builds on, memory as numbered slots is the bytes and addresses page, and a name standing for a value at one slot is the variables page. Neither is required to follow along here.

1A pointer is just a number, read as a place

A normal variable holds a value you want directly: the slot for count holds 7, and 7 is the answer. A pointer's slot holds a number too, say 4000, but you do not want 4000. You want whatever lives at address 4000.

The two look identical in memory. A slot holding the integer 4000 and a slot holding a pointer to address 4000 store the same number; nothing in the bits marks one as a place. The only difference is how your code reads it: as "the value I have" or as "where the value is." That is the whole trick, and the rest of the page is what it costs and what it buys.

2Following it is one hop

To get from the pointer to the real value, you follow the address it holds. You read the pointer, get a location, and go read what is at that location. One hop. That step has a name: dereferencing.

Python and JavaScript never make you write that hop, so it is easy to forget it happens. When you write user.name, the language quietly follows the reference to the object, then reads the field. You can make the hidden address show itself in Python with id(), which reports the address a name points at:

a = [1, 2, 3]
b = a
id(a) == id(b)   # True: two names, one address
b.append(4)
a                # [1, 2, 3, 4]: same list, reached through a

Two names, one address, so a change through either shows up through both. The hop is silent, but it is there, and it only works if the address points at a real value. A pointer can hold a special address meaning "points at nothing" (a null pointer), or an address whose value is gone (a dangling pointer); follow either and the same one hop lands on garbage or crashes. Those two failure modes get their own page: null and dangling pointers.

3Why go through an address at all

Holding a location instead of the thing sounds like extra work. The reason it is worth it, and the reason it bites you, are the same reason: two names can share one thing. Two pointers holding the same address both reach the same value, so a change through one shows up through the other. One thing, two ways in. Nearly every pointer bug you will hit is a case of this working when you did not expect it.

Sharing also means big things move cheaply. Hand a function a 10,000-item list and nothing is copied; the language passes the small address, and the function reaches your list through it. High-level languages do this for you by default, which is convenient and is exactly why the sharing can catch you off guard: you never asked for it.

Because a pointer is itself just a value in a slot, you can overwrite it, and two acts that look alike are not:

let a = { x: 1 };
let b = a;        // a and b point at one object
b.x = 99;         // change the thing:  a.x is now 99 too
b = { x: 0 };     // repoint b:  a is untouched, still { x: 99 }

The line b.x = 99 reaches through the arrow and edits the shared object, so a sees it. The line b = { x: 0 } only bends b's arrow to a new object; nothing at the old address moved, so a is untouched.

Repointing the arrow is not the same as changing what is at the end of it. Confusing the two is a classic bug, and it is the whole story behind what your language calls reference types: b = a copies the address, not the object.

4Copy vs share, and aliasing

The sharing bites hardest across a function call, where you feel safe handing an argument in. This is the case interviewers probe, and the one that breaks real code:

def mark_done(todo):
    todo["done"] = True   # edits the caller's dict, not a copy

mark_done(my_todo)        # my_todo["done"] is now True

You passed my_todo in and the function changed it, because the argument was a pointer to your dict, not a fresh copy. This is what people mean by Python's pass-by-object-reference, and it is why def f(lst): lst.append(1) grows the caller's list. When two names reach one thing this way, they are aliases, and a change through one silently changes what the other sees. That is the b.x = 99 surprise from the top of the page, grown up into an aliasing bug.

So under everything runs one question: does this code hold the thing, or a pointer to it, and if it is a pointer, who else holds the same address? That is the difference between copy and share. Copy gives each side its own thing; share hands the same address to more than one holder. When a value seems to change on its own, the first check in your own code is whether some other name is aliased to it: did you write b = a where you meant a copy, or pass a list into a function that kept it?

A pointer is indirection: one extra hop between a name and its value. Every copy-vs-share decision, and every aliasing bug, is really a question about that hop and who else is standing at the other end of it.

What a pointer is: indirection

A pointer is a value that holds an address: where a thing lives, not the thing itself. Reading it is one hop to the real value, a hop called indirection, and most confusing pointer bugs are moments when you forget the hop is there. Take a common one: you copy a variable into another, change the copy, and the original quietly changes too.

You may write Python or JavaScript and never type the word pointer. You lean on pointers in every line anyway. Here is the bug that shows it:

let a = { x: 1 };
let b = a;      // copy a into b, or so it looks
b.x = 99;       // now a.x is 99 too, though you never touched a

You changed b and a changed with it. That surprise, and a whole family of bugs like it, comes from one fact: when a variable holds an object, a list, a dict, an array, it does not hold that thing. It holds a pointer to it. The line b = a copied the pointer, so now two names point at one object, and a change through either name shows up through both.

The sharpest version of this shows up in interviews, when a function quietly changes its caller's data. That payoff is section 3. The first two sections build the small picture that makes it obvious, so it stops catching you off guard.

1A name holds an address, not the thing

Think of memory as a long row of numbered boxes. An address is just the number of one box, the way a house number names one house on a street. Sometimes a box holds the value you want directly. Often it holds an address instead: a number that says "the value is in box 4000, go look there." A box holding such a number is a pointer.

Python lets you see the address a name points at. id(x) reports it as a plain number:

a = [1, 2, 3]
b = a
id(a) == id(b)   # True: two names, one address
b.append(4)
a                # [1, 2, 3, 4]: same list, reached through a

The names a and b are two boxes holding the same address. The list lives somewhere else, in one place, and both names point at it:

Following the address, going from the name to the value it points at, is that one hop. Python and JavaScript take the hop for you every time: when you write a[0] or user.name, the language reads the pointer, lands on the object, then reads inside it. You never write the hop, so it is easy to forget it is there. The hop has a formal name, dereferencing, but the picture above is the whole idea.

2Changing the thing vs repointing the name

Because a name is itself a box holding a pointer, you can overwrite the pointer. Two lines that look almost the same do very different things:

let a = { x: 1 };
let b = a;        // a and b point at one object
b.x = 99;         // change the thing:   a.x is now 99 too
b = { x: 0 };     // repoint b:          a is untouched, still { x: 99 }

b.x = 99 follows b's arrow and edits the shared object, so a, which points at the same object, sees the change. b = { x: 0 } does not touch that object at all. It swings b's arrow to a brand new object and leaves a pointing where it always did.

Changing the thing the arrow points at is not the same as repointing the arrow. b = a copies the address, not the object; b.x = 99 reaches through the address and edits the one object both names share.

3Share, not copy: where the bugs live

The sharing bites hardest across a function call, exactly where you feel safe handing an argument in. This is the case interviewers probe, and the one that breaks real code:

def mark_done(todo):
    todo["done"] = True   # edits the caller's dict, not a copy

mark_done(my_todo)        # my_todo["done"] is now True

You passed my_todo in, and the function changed your dict, because the argument was a pointer to it, not a fresh copy. This is what people mean by Python's pass-by-object-reference: the function gets the same address you hold, so def f(lst): lst.append(1) grows the caller's list. Handing in a million-item list copies nothing, which is fast and convenient, and is the same fact behind the classic mutable-default-argument trap (def f(acc=[]) reuses one shared list across calls).

An alias is two names for one thing. When two names reach one object this way, they are aliases, and a change through either one silently changes what the other sees. So how do you get a real, independent copy instead? Ask for one:

b = a.copy()      # or list(a), dict(a): a new object, its own address
b.append(4)       # a is untouched now

One more thing can go wrong with the hop itself. A pointer can hold a special address meaning "points at nothing," which in Python is None (a null pointer). Follow it and you get AttributeError: 'NoneType' object has no attribute 'x', the "why is this None" crash. Python spares you the other classic, a dangling pointer, an address whose object is already gone: its garbage collector keeps an object alive as long as any name still points at it. Both failure modes get their own page, null and dangling pointers.

When a value seems to change on its own, run one check first: does another name hold the same address? Did you write b = a where you meant b = a.copy()? Did you hand a list to a function that kept it? Every copy-vs-share decision and every aliasing bug reduces to that one hop, and to who else is standing at the other end of it.

What is a runtime?

A runtime is the layer that actually runs your program, it takes the code you wrote and carries it out, step by step.

That sentence uses the word layer. Before it means anything, we need to build it up slowly. We will start with something simpler.

1Code is a representation

Code is not the thing itself. Code is a representation of something else.

Morse code is not sound, and it is not the message. It is a way to represent letters as short and long signals. Writing is not speech. It is a way to represent speech with marks on a page. Source code works the same way: it is a way for a person to write down instructions, structure, data, and intent in a form they can read.

When you write print("hello"), the computer is not reading Python the way you read a sentence. That line is just one representation. To do anything real, it has to be turned into other representations first. Those steps are done by layers, so let us look at what a layer is.

2What is a layer?

A layer is a way of working that hides the details below it and gives you a simpler set of things to work with.

You already use layers every day, without calling them that.

Driving a car

Ordering food

Writing code

You useprint("hello") Hidden from youhow letters reach the screen

In each case you work with a small, simple set of controls, and someone else's machinery underneath does the hard part. You do not control the spark timing in the engine. You do not tell the kitchen how to cook. You do not write the code that turns letters into dots of light.

So every layer does two things:

  • it hides the messy details below it, and
  • it gives the layer above a simpler set of words to work with.

3How you use a layer: the interface

If a layer hides what is below it, how do you actually use it?

You use its interface. An interface is the set of things a layer lets you ask for, plus a promise about what each one does. You use it without knowing what is behind it.

You have already seen three interfaces:

Driving a carinterface =the pedals and wheel
Ordering foodinterface =the menu
Writing codeinterface =print("hello")

Each interface is a short list of requests you are allowed to make. Press the pedal and the car speeds up. Ask for an item and the kitchen makes it. Call print and the text appears.

Now stack the layers. Each layer has its own interface. A request made at the top is translated downward, from one layer's interface into the next, until something physical happens.

print("hello"), you ask: "print this." You do not know how.

At each step, the upper layer asks in its own words, and the layer below translates the request into its own interface. Doing that translation is the real job of the layers in the middle, and the runtime is one of them.

You already know one kind of interface: an API (Application Programming Interface) is just the interface one program offers to another. The wider point is that interfaces are not only between programs. The pedal, the menu, and print are interfaces too, the same idea at every layer.

4A program is a tall stack of layers

You have now seen one layer and one interface. A whole program is many of them stacked up. At the top is a human purpose. At the bottom is a physical machine. Each layer is a representation of the one above it, in a form the layer below can use.

Read it from top to bottom. A person wants something. That want becomes requirements. The requirements become a description of how the product should behave. That becomes a design. The design becomes source code. The source code is turned into a form the runtime can run. Between each pair of layers is an interface, just like the ones you saw above.

5So where is the runtime?

The runtime is one of these layers. It sits in the lower-middle of the stack.

It is not the source code you wrote.
It is not the operating system.
It is not the CPU.

A runtime is the support environment that makes a program written in some language actually run. In the words we have been using:

A runtime is the layer your program runs on. It gives your code the interface it uses, and translates each request down toward the machine.

6What the runtime actually does

When you write Python, you use ideas like numbers, strings, lists, dicts, functions, classes, and errors. The CPU does not know any of these. The CPU only does very small, simple things. The runtime is the layer between these two worlds, and it translates from one to the other.

How does it actually do that? Let us look more closely at one runtime in particular.

7A closer look: the Python runtime

Let us make this concrete with a tiny program:

nums = [1, 2, 3]
print(sum(nums))

When you run it, the Python runtime does roughly this:

  • Compile it. Your text is turned into bytecode, a short list of very simple instructions the runtime can step through.
  • Run it, one step at a time. The runtime has a small loop (often called the virtual machine) that reads one bytecode instruction and does what it says, then moves to the next.
  • Make the objects. [1, 2, 3] becomes a list object in memory. The name nums is set to point at it.
  • Call the built-ins. sum and print are functions the runtime already provides. sum adds the numbers; print prepares the text.
  • Reach the operating system. To show the text, print asks the operating system to write it to the screen.
  • Clean up. When nums is no longer used, the runtime frees its memory. This is garbage collection.

Steps 1 and 2 are worth separating, because people often mix them up. They are two different actions.

Compiling (step 1)

source→bytecode

Changes the form. Representation to representation. Nothing runs yet.

Interpreting (step 2)

bytecode→does the work

Reads a form and carries it out. Representation to action.

So Python compiles first, then interprets. People who say "Python is interpreted" are only describing the second half.

Put together, here is what the Python runtime hands you:

a compilerturns your source into bytecode

Every one of these is something you would otherwise have to build yourself. The runtime provides them, so you can write nums = [1, 2, 3] and not think about any of it.

8Different languages, different runtimes

Other languages follow the same idea with different machinery.

Java: source → JVM bytecode → the JVM runs it (sometimes interpreting it, sometimes compiling it to machine code as it goes).
JavaScript in a browser: an engine like V8, plus what the browser provides, the page (the DOM), events, timers, and network requests.
Node.js: the same V8 engine, plus Node's own tools for files, networking, and processes.
C: a much thinner runtime. A C program can compile straight to machine code, but it still needs startup code, a way to call functions, memory setup, a standard library, and a way to talk to the operating system.

So a runtime is not always a big virtual machine. Some are thick, some are thin. But there is always some support layer present while the program runs.

9None of the layers is magic

This is the idea behind everything above: computing is built from layers, and each layer is built from the one below it. Nothing is magic, it only looks that way until you see the layer underneath.

a Python listis made oflower-level code and memory.
a function callis made ofstack frames and a few simple rules.
an erroris made ofrules for stopping and moving control somewhere else.
a web requestis made oflibraries, the runtime, the OS, drivers, hardware, and real signals on a wire.

10Programming is building with layers

It is easy to picture programming as "writing code", typing lines into a file. That is part of it. But most of the skill is something else: building a program out of layers, and using their interfaces well.

When you write Python, you are standing on the runtime's interface. The runtime stands on the operating system's interface. The operating system stands on the hardware. You are always working at one layer, using the interface of the layer below, and offering an interface to the layer above.

So on any task, a useful question is not only "what code do I write?" but also:

  • Which layer am I working in?
  • What interface am I using, and what does it promise?
  • What is hidden below me that I can rely on, and when do I need to look one layer deeper?

You do not need to understand every layer down to the physics. You can write a useful Python program without knowing how a transistor works. But the layer directly beneath you, the runtime, is worth understanding, because you stand on it every day. Knowing what it offers, and that none of it is magic, makes you better at the layer you actually work in.

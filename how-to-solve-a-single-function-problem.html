How to solve a single-function problem

A single-function problem is a programming interview problem you can solve with one function. They have names like rotate an N by N matrix or best time to buy and sell stock.

What makes these hard is doing them live: in front of an interviewer, thinking out loud, with the clock running.

A structured approach is what keeps you steady under that pressure, and the one here is built around a single rule:

Do not write code until you can state the method in plain words and predict what it will produce.

This is not a new idea. It is the oldest advice in the field, and it is what the practical intelligence and metacognition pages describe in the abstract: build a frame, then predict, then act. The five steps make it concrete.

1Understand it

Restate the problem in your own words. Pin down three things: what comes in (the type and shape of the input, and its limits), what goes out (the exact form of the answer), and what makes an answer correct.

Then ask about the cases the statement leaves unsaid. For "is this string a palindrome," does upper and lower case matter? Do spaces and punctuation count? Is the empty string a palindrome? You cannot solve a problem until you know all of its conditions and constraints, the ones stated and the ones you have to ask about.

2Write the cases

Work a few small inputs by hand, on paper, and produce the answers yourself: a normal case, and the awkward ones, the empty input, a single element, the largest case allowed.

Then write each one down as an input and its expected output. These are your test cases. The moment the function exists you run them as basic unit tests; until then they are the targets you are coding toward. If you cannot produce an answer by hand, you do not understand the problem yet, and no code will fix that.

3Write the method as a comment skeleton

Say how you would do it, in precise English, written as comments in the code, each comment small enough to become a line or two of code. Start with the simplest correct method, even if it is slow.

This is the step most people skip, and the one that does the most work. It turns the rough idea in your head into an explicit, ordered plan, and it leaves you a scaffold: the comments stay, and you fill code in beneath them.

4Implement against the skeleton

Fill in the code under each comment, one comment at a time. Each comment is a small, clear task, so you are never holding the whole problem in your head, only the one line you are turning into code.

If a comment is too vague to code directly, it is too vague. Break it into smaller comments until each one is obvious.

5Run the cases; predict, then check

Run the cases you wrote in step 2. Before each run, say what you expect the answer to be. Then compare.

If the result matches your prediction, you understand your own code. If it does not, the gap between expected and actual is the bug, and the comment above the offending code tells you roughly where to look. A run with no prediction teaches you almost nothing; a run against a prediction is a test. The edges are already covered, because you wrote them as cases.

A worked example

The problem: given a string, return whether it reads the same forwards and backwards.

Understand it. Input: a string. Output: true or false. We decide that characters are compared exactly (case matters, spaces count), and that the empty string counts as a palindrome.

Write the cases. A normal case and the awkward ones, written as input and expected output:

is_palindrome("racecar") -> true
is_palindrome("hello")   -> false
is_palindrome("")        -> true
is_palindrome("a")       -> true

Write the method as a comment skeleton. Walk in from both ends at once, written as comments with no code yet:

function is_palindrome(s):
    # start one pointer at the left end, one at the right end
    # while the pointers have not met:
    #   if the two characters differ, return false
    #   step both pointers inward
    # the pointers met with no mismatch, so return true

Implement against the skeleton. Fill in the code under each comment; the comments stay as a guide:

function is_palindrome(s):
    # start one pointer at the left end, one at the right end
    left  = 0
    right = length(s) - 1
    # while the pointers have not met:
    while left < right:
        # if the two characters differ, return false
        if s[left] != s[right]:
            return false
        # step both pointers inward
        left  = left + 1
        right = right - 1
    # the pointers met with no mismatch, so return true
    return true

Run the cases; predict, then check. On "racecar" I expect true: every pair matches until the pointers meet. On "hello" I expect false: the first pair, h and o, differs. On "" and "a" I expect true: the pointers start already met, so the loop never runs. Running the four cases gives exactly those. Done.

When the basic method runs out

This is enough for most single-function problems. Harder ones add more: writing the function's signature and a one-line purpose before anything else; letting the shape of the data drive the shape of the code; matching the problem to a pattern you have solved before; choosing a data structure on purpose; reasoning about why the code is correct rather than only testing it; and, when you are stuck, stepping away from the keyboard so the back of your mind can work.

Those build on the five steps here. They do not replace them.

References

The five steps are the common core of a long heritage of problem-solving advice.

  • George Polya, How to Solve It (1945): understand, plan, carry out, look back.
  • Matthias Felleisen and others, How to Design Programs: the design recipe, examples and a template you fill in, with the examples turned into tests.
  • Niklaus Wirth, "Program Development by Stepwise Refinement" (1971): grow code by refining a sketch into detail, which is the comment skeleton.
  • Leslie Lamport, "Thinking Above the Code": write the method down before you code.
  • Jon Bentley, Programming Pearls: getting the problem definition right matters as much as the algorithm.
  • Named interview variants that teach the same core: REACTO (Codesmith and others), UMPIRE (CodePath).

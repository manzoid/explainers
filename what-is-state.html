What is state?

At any instant, a running program is holding some information: the values worked out so far, and where it is in the job. That held information is its state. Running the program means changing that state, step by step, from the input toward the answer.

Here is a bug you have hit. You compute a total from a list, save it, and later something appends to that list. Now the saved total is wrong, even though the line that set it never ran again and every line still looks correct. Nothing in the code is broken. What broke is the state: the information the program held stopped matching what you believed about it. Almost every hard bug is a version of this, and that is why state is worth understanding on its own.

This page is about that held information: what it is, how it changes, and why it causes most of the trouble in programs. The CPU page covered the machine that does the work; here we look at what it works on.

1State is the information a program holds right now

State is the data a program carries from one moment to the next, the data that shapes what it does next. A plain calculation takes an input, returns an answer, and keeps nothing between calls. That kind of function has no state. A program has state when what it does depends on what came before.

Physically, that information is bits sitting somewhere: in memory, or in the small set of fast cells inside the CPU called registers. Bits alone mean nothing until the program reads them a certain way. The same eight bits can stand for the number 65 or the text character 'A'; nothing in the bits says which, the program decides. State is always bits plus a reading of them. The memory those bits sit in gets its own page next; here we only need the idea that the state is real, and it is somewhere.

2State changes over time

Freeze the program at one instant and you get a snapshot: one whole arrangement of bits. Take a snapshot at every instant and play them in order, and you have the entire run. Each step, the machine reads the current arrangement and writes a slightly changed one. Fetch, decode, execute, from the CPU page, is exactly the rule for how one snapshot becomes the next.

Here is a tiny program that adds up a list. Its whole logical state is the two values total and i:

nums = [2, 3]
total = 0
for i in range(len(nums)):
    total = total + nums[i]

The run is just this sequence of arrangements. Any single snapshot tells you where the program is; the sequence tells you what it is doing.

3A running program is a state machine

Put the snapshot and the sequence together and you have a state machine: at any moment the program is in one state, one whole arrangement, and each step takes it to the next. Every running program is one, planned or not. When its states change in an order you did not expect, you get a bug: a form that submits before the password is set, a cart that checks out twice because two clicks both saw the same "not yet paid" state.

Because that unplanned state machine is where those bugs come from, people often flip the idea around and design one on purpose: name a small set of allowed states, like loading, ready, and error, and permit only certain steps between them. That deliberate version is a design tool, and it gets its own page.

4Changing state is literal

To change state is to overwrite the bits that held the old value. The old value is simply gone; the cell now holds the new one. That overwrite has a name: a mutation. This is the same word Python uses when it calls a list mutable and a tuple immutable. A mutable object is one whose bits the language lets you overwrite in place.

Computation is state, mutated forward: start at the arrangement that stands for the input, apply many small mutations, and arrive at the arrangement that stands for the answer.

Seeing state as one real arrangement of bits explains a bug that catches everyone once. Two names can point at the same list. Mutate through one, and the other "changes" too, because there is only one arrangement and both names read it:

a = [1, 2]
b = a          # b is not a copy; it points at the same list
b.append(3)
print(a)       # [1, 2, 3], not [1, 2]

Nothing touched a by name, yet a changed. How two names can share one piece of state, and when you get a copy instead, is its own page.

5Two kinds of state, and the one you debug with

The machine's state is every live bit: all of memory, every register, and the program counter that says which instruction is next. It changes on every single step; even a step that does no real work still moves the program counter forward.

The program's logical state is the part you actually reason about: the counter, the list, the cart, the balance. It changes on many steps, not all. Most of the time you think in the logical state and ignore the rest. This is also the state you manage by hand in application code: React's useState, a Redux store, the fields on an object. Same idea as here, one level up, further from the bits.

Then something breaks, and the logical view is no help, because the idea you had about the state is the thing that is wrong. That is when you drop down to the machine's view and read the actual values, one at a time, to find where your idea of the state and the real state parted ways.

6Why state is where the trouble lives

State is the one thing in a program that does not hold still, and that is exactly what makes it hard. The values you are trying to reason about keep changing while you reason about them.

Most hard bugs are a wrong belief about the state: what it is right now, or what some step changed it into.

This is the bug from the top of the page, made concrete:

items = [2, 3]
total = sum(items)   # total is 5; you believe total tracks items
items.append(10)     # a later step changes items...
                     # ...but nothing recomputes total
print(total)         # 5   -- the value you are holding
print(sum(items))    # 15  -- what it should be now

Every line is correct on its own. The bug is the belief that total still reflected items after the list changed. You will meet the same shape everywhere: a cached count that another step made stale, a numpy view you thought was a copy, a variable you were sure held one thing holding another. Debugging is largely the work of finding where your belief about the state and the real state came apart. It is the same difficulty that shows up in hard problems everywhere, not only in code: reasoning about something that keeps changing under you. The page on why hard problems are hard goes into that.

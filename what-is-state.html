What is state?

You compute a total from a list, save it, then later some code appends to the list. Now the saved total is wrong, even though the line that set it never ran again and every line still looks correct. Nothing in the code is broken. What broke is the program's state.

State is the information a program is holding right now: the values worked out so far, and where it is in the job. Running a program means changing that state, step by step, from the input toward the answer. Almost every hard bug is a moment like the one above, where the state stopped matching what you believed about it. That is why state is worth understanding on its own, and it is one of the first things an interviewer probes when they ask you to trace a piece of code.

This page is about that held information: what it is, how it changes, and why it causes most of the trouble in programs.

1State is the information a program holds right now

State is the data a program carries from one moment to the next, the data that shapes what it does next. A plain calculation takes an input, returns an answer, and keeps nothing in between. That kind of function has no state. A program has state when what it does depends on what came before.

Here is the smallest example. This loop adds up a list, and at any instant its whole state is two values, total and i:

nums = [2, 3]
total = 0
for i in range(len(nums)):
    total = total + nums[i]

total holds the running answer; i holds where the loop is. Read those two and you know exactly where the program stands. Physically they are bits sitting in memory, which gets its own page; here all we need is that the state is real values you can point at.

2State changes over time

Freeze the program at one instant and you get a snapshot: one whole arrangement of values. Take a snapshot at every step and play them in order, and you have the entire run. Each step, the machine reads the current arrangement and writes a slightly changed one.

The run is just this sequence of arrangements. Any single snapshot tells you where the program is; the sequence tells you what it is doing.

3A running program is a state machine

A user double-clicks Buy. Two clicks fire, both read the same "not yet paid" state, and the order is placed twice. Nothing was mistyped. The program simply passed through a state you never pictured. That double charge happened because "paying" was never a state of its own: the code knew only "not paid" and "paid", so the second click still saw "not paid" and went ahead.

Line up every program and it works this way. At any moment it is in one state, one whole arrangement, and each step takes it to the next. That is a state machine, and every running program is one, whether you planned it or not. So people name the states on purpose. Give a checkout form a few named states and allow only certain steps between them:

empty  -->  filled  -->  submitting  -->  done

# "Buy" is allowed only when the state is filled, and it moves you to submitting
# a second click arrives while the state is already submitting, so it is refused

Naming submitting as a state of its own is what blocks the second click: once you are in it, another Buy is not an allowed step. Designing systems around named states like this is its own page.

4Changing state is literal

To change state is to overwrite the bits that held the old value. The old value is simply gone; the cell now holds the new one. That overwrite has a name: a mutation. It is the same word Python uses when it calls a list mutable and a tuple immutable. A mutable object is one whose contents the language lets you overwrite in place.

Computation is state, mutated forward: start at the arrangement that stands for the input, apply many small mutations, and arrive at the arrangement that stands for the answer.

Seeing state as one real arrangement explains a bug that catches everyone once. Two names can point at the same list. Mutate through one, and the other "changes" too, because there is only one list and both names read it:

a = [1, 2]
b = a          # b is not a copy; it points at the same list
b.append(3)
print(a)       # [1, 2, 3], not [1, 2]

Nothing touched a by name, yet a changed. How names refer to values, and when you get a copy instead of a shared one, are their own pages.

5Two levels of state, and the one you debug with

There are two levels to a program's state, and you use them at different times. The machine's state is every live value: all of memory, plus a program counter that marks which line runs next. In the loop from section 2, that is not only total and i but also the fact that, say, line 4 is up next.

The logical state is the slice you actually reason about: here, just total and i. Most of the time you think in the logical state and ignore the rest. This is also the state you manage by hand in application code. When a front-end framework talks about "state", React's useState or a Redux store, it means exactly this level: the logical state, named and tracked for you.

Then a bug appears and the logical view stops helping, because your idea of the state is the thing that is wrong. That is when you drop to the machine's level: stop reasoning in your head and read the real values one at a time, with a print or a debugger, until you find where your idea of the state and the real state split apart.

6Why state is where the trouble lives

State is the one thing in a program that does not hold still, and that is exactly what makes it hard. The values you are trying to reason about keep changing while you reason about them.

Most hard bugs are a wrong belief about the state: what it is right now, or what some step changed it into.

The stale total at the top of the page is one shape of this. Here is a different shape, where the state hides in a place you did not think had any:

def add_item(x, bucket=[]):
    bucket.append(x)
    return bucket

print(add_item(1))   # [1]
print(add_item(2))   # [1, 2]   -- you expected [2]

You expected a fresh list on every call. But the default list is made once, when the function is defined, and every call that leaves out bucket shares that one list. The state you forgot existed is the default itself. Same lesson as the stale total, different hiding place: the trouble was a belief about the state that did not match what was really there. Debugging is largely the work of finding where your belief about the state and the real state came apart.

7The point

State is just the information a program is holding right now. It changes every step, and a program is the sequence of those changes. Almost every hard bug is the same story: the state moved, and your picture of it did not. Get in the habit of asking "what is the state right now, and what did that step change it into", and most mysterious bugs stop being mysterious.

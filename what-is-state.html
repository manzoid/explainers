What is state?

At any instant, a running program is holding some information: the values worked out so far, and where it is in the job. That held information is its state. Running the program means changing that state, step by step, from the input toward the answer.

A running program is always moving information around. This page is about that information: what it is, and why it causes most of the trouble in programs. The CPU page covered the machine that does the moving; here we look at what it moves.

If you already know that variables hold values and that programs change those values over time, the first few sections will feel slow. The part that pays off is section 6 onward: the split between the machine's state and the state you reason about, and why state is where most bugs live.

1State is the information a program holds right now

State is the data a program carries from one moment to the next, the data that shapes what it does next. A plain calculation takes an input and returns an answer. It keeps nothing between calls. That kind of function has no state. A program has state when what it does depends on what came before.

Physically, that information is bits, sitting somewhere: in memory, or in the small number of fast storage cells inside the CPU called registers. The memory those bits sit in gets its own page next; here we only need the idea that the state is real, and it is somewhere.

2A snapshot means nothing on its own

Freeze the program at a single instant and look at memory. You see one arrangement of bits. By itself it is just a pattern, no more meaningful than a page of random digits.

It becomes state only under the program's interpretation: this region is the counter, that region is the list, this one is where we are in the loop. The same bits could mean anything. Take four bytes holding the value C0 A8 01 01. Read as a whole number, that is 3,232,235,777. Read as an IP address, it is 192.168.1.1. Read as a decimal (floating-point) number, it is something else again. Nothing in the bits says which; the program is what decides. State is always bits plus a reading of them.

3The film: state changes over time

Take a snapshot at every instant and play them in order, and you have the whole run. Each step, the machine reads the current arrangement and writes a slightly changed one. Fetch, decode, execute, from the CPU page, is exactly the rule for how one snapshot becomes the next.

Here is a tiny program that adds up a list. Its whole state is the two values total and i:

nums = [2, 3]
total = 0
for i in range(len(nums)):
    total = total + nums[i]

4A running program is a state machine

Put the snapshot and the film together and you have a state machine: at any moment the program is in one state, one whole arrangement, and each step takes it to the next. The CPU is the part that takes the step; the program, the instructions, decides which step to take.

This is the state machine you cannot avoid: every running program is one, planned or not. When its states change in an order you did not expect, you get a bug: a form that submits before the password is set, a cart that checks out twice. So many people flip the idea around and use it on purpose. They name a small set of states up front, say loading, ready, and error, and allow only certain steps between them. That deliberate version is a design tool, not a fact about the hardware, and it exists to keep the unplanned state machine from surprising you. It gets its own page later in the series.

5Changing state is literal

To change state is to overwrite the bits that held the old value. The old value is simply gone; the cell now holds the new one. That overwrite has a name: a mutation. This is the same word Python uses when it calls a list mutable and a tuple immutable. A mutable object is one whose bits the language lets you overwrite in place.

Computation is state, mutated forward: start at the arrangement that stands for the input, apply many small mutations, and arrive at the arrangement that stands for the answer.

Say that plainly, because higher up in a program a lot of effort goes into pretending state does not change in place. That is worth its own discussion later; underneath all of it, the bits are still being overwritten.

6Two kinds of state, and the one you debug with

The machine's state is every live bit: all of memory, every register, and the program counter that says which instruction is next. It changes on every single step; even a step that does no real work still moves the program counter.

The program's logical state is the part you actually reason about: the counter, the list, the cart, the balance. It changes on many steps, not all. Most of the time you think in the logical state and ignore the rest. This is also the state you manage by hand in application code: React's useState, a Redux store, the fields on an object. Same idea as here, one level up, further from the bits.

Then something breaks, and the logical view is no help, because the idea you had about the state is the thing that is wrong. That is when you drop down to the machine's view and read the actual bits, register by register, to find where your idea of the state and the real state parted ways.

7Why state is where the trouble lives

State is the one thing in a program that does not hold still, and that is exactly what makes it hard. The values you are trying to reason about keep changing while you reason about them.

Most hard bugs are a wrong belief about the state: what it is right now, or what some step changed it into.

You have chased one of these. A screen shows the wrong number because it read a cached value that another step had already made stale. The code looks right at every line; the belief about the state is what was wrong. This is one reason debugging is so hard. It is the same difficulty that shows up in hard problems everywhere, not only in code: reasoning about something that keeps changing under you. The page on why hard problems are hard goes into that.

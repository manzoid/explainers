State machines

A state machine is a short list of named states, plus the transitions allowed between them, with exactly one state active right now. You draw the states a thing can be in, draw the arrows it may follow, and any arrow you did not draw simply cannot happen.

The state page showed that every running program is already a state machine whether you planned it or not: at each step it sits in one whole arrangement and moves to the next. The CPU underneath works the same way, fetch then decode then execute, a small cycle of states. This page is about the deliberate version: choosing on purpose to model a small piece of your program as a handful of named states you write down yourself. That choice is one of the cheapest ways to make a whole class of bugs impossible to write.

1Three parts, that is all

A state machine has just three parts. A set of states, each with a name. A set of transitions, the arrows that say which state you may move to from which. And one current state, the single state the thing is in at this moment.

The word "finite" you sometimes see attached to it, as in finite state machine, only means the list of states is short and fixed. You can write them all down on a napkin. That is the whole point: a small, closed set you can hold in your head.

2A form that submits

Think about a form that submits to a server. It is idle until the user hits submit. Then it is loading while the request is out. Then it lands on success or error. Four states, and the arrows between them are obvious once you name them.

The same shape fits more than screens. An order goes cart, then paid, then shipped, then delivered, and the arrows only run forward. Different words, one idea underneath.

3One variable instead of five booleans

Here is what you buy. Look at the form again, but built the way most of us build it the first time, as a bag of booleans:

let isLoading = false
let isSuccess = false
let isError   = false

Three true-or-false flags have eight combinations. But most of them are nonsense. isLoading and isError both true means the request is somehow in flight and already failed. isSuccess and isError both true means it worked and did not. These states have no meaning, yet the code lets you write them, so sooner or later some path does, and you get a spinner sitting on top of an error message.

Now say it as one state instead:

state = "idle"   // or "loading", "success", "error"
Model the valid states and transitions, and the illegal states stop existing. There is no arrangement of the code that puts you in "loading and error at once," because there is only ever one state, and "loading and error" is not one of the names.

The bug did not get caught. It got deleted. You cannot fix a bug that the shape of your data does not let you write.

4Transitions you did not allow cannot happen

The states are half of it. The arrows are the other half, and they carry the same protection. In the order example, the only way out of cart is to paid. There is no arrow from cart straight to shipped. So "we shipped an order nobody paid for" is not a bug you have to remember to guard against. It is a transition the machine does not offer.

When your code is a pile of flags and if statements, every function is free to set any flag at any time, and the allowed order of events lives only in your head and your hope. A state machine writes that order down. The current state decides which transitions are even on the table; ask for one that is not, and the machine refuses instead of quietly doing something wrong.

You can write that down as a plain table: for each state, the events it accepts and where each one leads. The form from section 2 is five lines.

const transitions = {
  idle:    { submit: "loading" },
  loading: { ok: "success", fail: "error" },
  error:   { retry: "loading" },
  success: { reset: "idle" },
}

function next(state, event) {
  const to = transitions[state][event]
  return to ?? state   // event not allowed here: stay put
}

That is the whole machine. An event that the current state does not list has nowhere to go, so it does nothing, and "we shipped before paying" cannot run because the arrow is not in the table. In React the same idea is useReducer, where the reducer is this next function; when the states and events grow past a napkin, a library like XState handles the same table for you.

5How to notice one hiding in your code

You rarely set out to build a state machine. More often you find one already tangled up in your code, asking to be pulled out. A few smells give it away.

Several booleans that describe the same one thing, where only some combinations make sense. A comment like // note: never true while loading. A cluster of if checks that keep testing the same flags in different orders. A status field that is really a word from a fixed list, "pending", "active", "closed", guarded by scattered rules about what may follow what.

When you see that, try the swap. Write down the handful of states the thing can actually be in. Draw the arrows. If the drawing is small and closed, you found a state machine, and folding those booleans into one state variable will usually delete code and bugs at the same time.

State machines

A state machine is a short list of named states with arrows between them, and any arrow you did not draw simply cannot happen.

The state page showed that every running program is already a state machine whether you planned it or not: at each step it sits in one whole arrangement and moves to the next. This page is about the deliberate version: choosing on purpose to model a small piece of your program as a handful of named states you write down yourself. That choice is one of the cheapest ways to make a whole class of bugs impossible to write.

1Three parts, that is all

A state machine has three parts. A set of states, each with a name. A set of transitions, the arrows that say which state you may move to from which. And one current state: a single variable holding the state the thing is in right now. That variable is the machine's whole memory. Everything the code does next depends on it.

2A form that submits

Think about a form that submits to a server. It is idle until the user hits submit. Then it is loading while the request is out. Then it lands on success or error. Four states, and the arrows between them are obvious once you name them.

The same shape fits far more than screens. Anything that moves through a fixed set of stages one at a time, an order, a support ticket, a user's login flow, is a state machine waiting to be named.

3One variable instead of three booleans

Here is what you buy. Look at the form again, but built the way most of us build it the first time, as a bag of booleans:

let isLoading = false
let isSuccess = false
let isError   = false

Three true-or-false flags have eight combinations. But most of them are nonsense. isLoading and isError both true means the request is somehow in flight and already failed. isSuccess and isError both true means it worked and did not. These states have no meaning, yet the code lets you write them, so sooner or later some path does, and you get a spinner sitting on top of an error message.

Now say it as one state instead:

let state = "idle"   // or "loading", "success", "error"
Model the valid states and transitions, and the illegal states stop existing. There is no arrangement of the code that puts you in "loading and error at once," because there is only ever one state, and "loading and error" is not one of the names.

The bug did not get caught. It got deleted. You cannot fix a bug that the shape of your data does not let you write.

This is also a pattern interviewers reach for. String tokenizing, parsing, and simple regex problems are often state machines in disguise: you read one character at a time and step between a few named states. Naming the states out loud is often the whole method the interviewer is waiting to hear.

4Transitions you did not allow cannot happen

The states are half of it. The arrows are the other half, and they carry the same protection. Take an order: it goes cart, then paid, then shipped, then delivered, and the only way out of cart is to paid. There is no arrow from cart straight to shipped. So "we shipped an order nobody paid for" is not a bug you have to remember to guard against. It is a transition the machine does not offer.

When your code is a pile of flags and if statements, every function is free to set any flag at any time, and the allowed order of events lives only in your head and your hope. A state machine writes that order down. The current state decides which transitions are even on the table; ask for one that is not, and the machine refuses instead of quietly doing something wrong.

You can write that down as a plain table: rows are states, columns are events, and each cell is the state you land in. The blank cells are the transitions that do not exist, the ones that simply cannot happen.

The same table, written as data your code can read:

const transitions = {
  idle:    { submit: "loading" },
  loading: { ok: "success", fail: "error" },
  error:   { retry: "loading" },
  success: { reset: "idle" },
}

// next(state, event) is a reducer: current state and an event go in,
// the next state comes out. In React this is exactly useReducer.
function next(state, event) {
  const to = transitions[state][event]
  // ?? means: if there is no arrow for this event, keep the current state
  return to ?? state
}

next("idle", "submit")  // "loading"
next("idle", "ok")      // "idle"   -- no such arrow, so the event is ignored

That is the whole machine. An event the current state does not list has nowhere to go, so it does nothing, as the last line shows: asking idle for ok just returns idle. The table is also data, so you can log it, test it, print it, or swap it at runtime; a pile of if statements can do none of that. And this shape is what interviewers probe in system-design questions too, order lifecycles, auth flows, retry logic, where the real question is which transitions you allow.

In React the next function above is what you pass to useReducer. When the machine grows past what fits on a napkin, say more than six or seven states, or transitions that need side effects like starting a timer or firing a request, a library like XState keeps the same table but adds those side effects, nested states, and a visual diagram the plain object cannot give you.

5How to notice one hiding in your code

You rarely set out to build a state machine. More often you find one already tangled up in your code, asking to be pulled out. A few smells give it away.

  • Several booleans that describe the same one thing, where only some combinations make sense.
  • A comment like // note: never true while loading that guards a combination the types allow but the logic forbids.
  • A cluster of if checks that keep testing the same flags in different orders.
  • A status field that is really a word from a fixed list, "pending", "active", "closed", guarded by scattered rules about what may follow what.

When you see that, try the swap. Take the tangled flags from section 3 and fold them into one state variable:

Before

let isLoading = false
let isError   = false
let isDone    = false
// never true together...

if (isLoading) showSpinner()
if (isError)   showError()
if (isDone)    showResult()

After

let status = "idle"
// "loading" | "error" | "done"




if (status === "loading") showSpinner()
if (status === "error")   showError()
if (status === "done")    showResult()

The three flags become one word, and the impossible combinations, loading and done at once, cannot be written. If the drawing of states and arrows is small and closed, you found a state machine, and folding those booleans into one state variable will usually delete code and bugs at the same time.

6The point

You now have a way to turn a class of bugs into shapes that cannot be built: name the states, draw the arrows, keep one current state. That is a tool you can use tomorrow in your own code, and a pattern to reach for in interviews, where it shows up under names like finite state machine, event-driven design, and reducer. When a problem is really about a thing moving through stages one at a time, saying so out loud is often half the answer.

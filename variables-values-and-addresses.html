Variables, values, and addresses

A variable is really three things held apart: the name you wrote, the place in memory it stands for, and the value sitting there right now. Take the line you just typed: you wrote total, not address 4096. Keep the name, the place, and the value separate, and one confusing mechanism becomes plain, the one behind two bugs you have surely hit.

The two bugs: aliasing, where two names quietly share one object, so editing through one changes the other; and unexpected mutation, where a function changes data you never handed it to change. They feel like separate mysteries. This page shows they are the same thing seen from two angles, and that the fix falls out once you split every line into a name, a place, and a value. It is also standard interview ground: when an interviewer asks whether Python passes by value or by reference, or why editing a copy also edited the original, they are checking that you hold these three apart.

1Three things, not one

Look at one line: score = 10. It is easy to read that as a single thing, a box called score with 10 in it. But there are three separate things here, and splitting them apart is the whole point of the page.

There is the name, score, which you wrote. There is the place, a spot in memory with its own address, which the machine picked. And there is the value, 10, the bits now sitting in that place. The name is yours. The place is fixed for the life of the variable. The value can change.

The machine does not know the word score. It works in addresses: read the cell at this address, write to the cell at that one. Your language keeps a table that says this name means that address, so everywhere you wrote score it uses the real address instead. You think in names; the machine runs on addresses; the table connects the two. That is why one value can sit in memory under one name, two names, or none.

2The place, drawn

Before the diagram, fix the one thing that trips people up: the equals sign. In score = 42 it is not a claim that two things are equal.

Read = as "store into", not "is equal to". score = 42 means: take the value on the right and store it into the place named on the left.

With that in hand the diagram is obvious. On the left is the state after score = 10: the name score sits above its cell, and the table in the middle records that score means address 4096. On the right, after score = 42 runs, only the value in the cell changed. Same name, same address, new value.

3Two names, one place

Once the name and the place are separate, a sharp question appears: can two names point at the same place? Yes. That is where the trouble starts. Here it is in JavaScript. You put an object in a, copy a into b, and change something through b:

// a and b end up naming the same object
let a = { count: 0 };
let b = a;        // not a fresh copy, a second name
b.count = 5;      // change it through b
a.count;          // 5  -- a changed too

Nothing touched a on that third line, yet a.count is now 5. The reason: the value stored in a was not the object itself, it was the address of the object, which lives elsewhere in memory. let b = a copied that address, so both cells now hold directions to the one object. Changing it through b changes the same object you can also reach through a.

Python works the same way. b = a on a list or dict makes b is a true, and editing through one edits the other. But the rule does not hold for plain numbers. Watch: a = 0, then b = a, then b = 5, and a is still 0. The difference is what you can change in place. Numbers, strings, and booleans are immutable: you can never edit one in place, so the language is free to give each name its own value and sharing never bites. Lists, dicts, and objects are mutable: a shared one shows every edit through every name. Whether an assignment copies the value into a fresh place or just adds a second name to the old place is the copy versus share question, and it decides a large share of real bugs.

A name is not the thing. It is a label on a place. Two labels can sit on one place, and then changing the object through "either" name changes the one thing underneath.

4The function that changed your list

A function that "changed something it shouldn't have" is this same mechanism, one step removed. Passing an argument is an assignment: the parameter becomes a second name for whatever you handed in. If you hand in a list, the function does not get a copy, it gets another name for your list.

# Python
def add_tag(tags):
    tags.append("new")   # edits the caller's list

my_tags = ["a", "b"]
add_tag(my_tags)
my_tags                  # ["a", "b", "new"] -- you never appended

Inside add_tag, the name tags points at the same list as my_tags outside. The append changes that one list, so the caller sees the extra item. This is not a new bug. It is aliasing from section 3, now crossing a function boundary: the same two names on one place, so it bites in the same way.

The fix is to make a real copy before you change it, so the function works on its own place:

# Python: copy first
def add_tag(tags):
    tags = tags.copy()   # a fresh list, its own place
    tags.append("new")
    return tags

// JavaScript: same idea
let b = { ...a };        // a fresh object, not a shared address

Notice what tags = tags.copy() does in terms of the three parts. It does not change the caller's list. It points the name tags at a brand new place, so every later line touches only that copy. Changing the value in a place is what leaks back to the caller; pointing a name at a different place does not. A shallow copy like this is enough when the items themselves are not shared objects you also mutate; when they are, you need a deeper copy.

5The point

When a line surprises you, do not guess. Split it into the three parts and ask, in order:

  • What place does this name refer to?
  • Who else holds a name for that same place?
  • Did this line change the value in that place, or point the name at a different place?

Changing the value in a shared place is what shows up under every other name; pointing a name at a fresh place does not. Keep name, place, and value separate, and most mutation bugs stop being mysteries and start having an obvious explanation.

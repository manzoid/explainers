Variables, values, and addresses

A variable is a name for a place in memory. The value is whatever sits in that place right now. Keep the name, the place, and the value apart, and two puzzles get simple: two names that accidentally share one cell, and why your function changed something it was never supposed to touch.

The memory page showed raw memory: a long row of numbered byte-cells, nothing but bits and their addresses. But you do not write address 4096 in your code, you write total. This page is the bridge. It shows how the human names in your code map down to those numbered cells, and why that mapping is behind copy versus share and pointers, the two ideas that follow.

1Three things, not one

Look at one line of code: score = 10. It is easy to read that as a single thing, a box called score with 10 in it. But there are three separate things here, and untangling them is the whole point of this page.

There is the name, score, which you wrote. There is the place, a spot in memory with its own address, which the machine picked. And there is the value, 10, the bits now sitting in that place. The name is yours. The place is fixed for the life of the variable. The value can change.

2The name is for you; the machine runs on addresses

The machine does not know the word score. Addresses are what it works with: read the cell at this address, write to the cell at that one. When you compile or run the program, the tool builds a table that says this name means that address, and everywhere you wrote score it substitutes the real address.

So the name is a label the tool translates away. You think in names; the machine runs on addresses; the table connects them. This is why the same value can sit in memory under one name, two names, or none: the value lives in the place, and names are just handles you attached to it.

3The place, drawn

On the left of the diagram, the name score floats above its cell. The dashed line is the table connection: the name up top, the place at address 4096 below, the value inside. To the right, the same cell after score = 42 runs. Same name, same address, new value.

That gap, a fixed place holding a value that is free to change, is where the next idea comes from. It is the same overwrite the state page called a mutation, now seen from the language level.

4Two names, one place

Once the name and the place are separate, a sharp question appears: can two names point at the same place? Yes. And that is where the trouble starts.

Here is the case that bites most people, in JavaScript. You put an object in a, then copy a into b, and change something through b:

// a and b end up naming the same object
let a = { count: 0 };
let b = a;        // not a fresh copy, a second name
b.count = 5;      // change it through b
a.count;          // 5  -- a changed too

Nothing touched a on that third line, yet a.count is now 5. The reason: the value stored in a was not the object itself, it was the address of the object elsewhere in memory. let b = a copied that address, so both cells hold directions to the one object. A value that is the address of another place is a pointer, and it is what lets two names share one thing.

A name is not the thing. It is a label on a place. Two labels can sit on one place, and then changing "either" changes the one thing underneath.

Whether an assignment copies the value into a fresh place or just adds a second name to the old place is the copy versus share question, and it decides a large share of real bugs. A function that "changed something it shouldn't have" is almost always this: it was handed a name, not a copy.

5Three questions when something surprises you

When a line does something you did not expect, pull the name, the place, and the value apart and ask three plain questions. Run them on the case above:

What place does this name refer to? b refers to the same place as a, because let b = a copied a shared address. What value is in that place right now? One object, whose count is now 5. Did this line change the value, or point a name at a different place? b.count = 5 changed the value in the shared place; it did not hand a or b a new place.

Hold those three apart and pointers, sharing, and the bugs that come with them stop being mysterious.

Reuse and share freely.

Variables, values, and addresses

A variable is a name for a place in memory. The value is whatever sits in that place right now. Two bugs stop being mysterious once you keep the name, the place, and the value apart: two names that accidentally share one object, and a function that changed something it was never handed to change.

You write total, not address 4096. This page shows how the human name in your code maps down to a numbered place in memory, and why that mapping is behind a whole family of bugs you have probably already hit. It is also standard interview ground: when an interviewer asks whether Python passes by value or by reference, or why editing a copy also edited the original, they are checking that you hold exactly these three things apart.

1Three things, not one

Look at one line: score = 10. It is easy to read that as a single thing, a box called score with 10 in it. But there are three separate things here, and untangling them is the whole point of the page.

There is the name, score, which you wrote. There is the place, a spot in memory with its own address, which the machine picked. And there is the value, 10, the bits now sitting in that place. The name is yours. The place is fixed for the life of the variable. The value can change.

The machine does not know the word score. It works in addresses: read the cell at this address, write to the cell at that one. Your language keeps a table that says this name means that address, so everywhere you wrote score it uses the real address instead. You think in names; the machine runs on addresses; the table connects the two. That is why one value can sit in memory under one name, two names, or none.

2The place, drawn

In the diagram, the name score floats above its cell. The dashed line is the table connection: the name up top, the place at address 4096 below, the value inside. To the right, the same cell after score = 42 runs. Same name, same address, new value.

This is where the equals sign trips people up. In score = 42 it is not a claim that two things are equal. It is an instruction: take the value on the right and store it into the place named on the left. Read it as "store into", not "is equal to". Then the diagram is obvious: only the value in the cell changes; the place keeps its address.

3Two names, one place

Once the name and the place are separate, a sharp question appears: can two names point at the same place? Yes. That is where the trouble starts. Here it is in JavaScript. You put an object in a, copy a into b, and change something through b:

// a and b end up naming the same object
let a = { count: 0 };
let b = a;        // not a fresh copy, a second name
b.count = 5;      // change it through b
a.count;          // 5  -- a changed too

Nothing touched a on that third line, yet a.count is now 5. The reason: the value stored in a was not the object itself, it was the address of the object, which lives elsewhere in memory. let b = a copied that address, so both cells now hold directions to the one object. Changing it through b changes the same object you can also reach through a.

Python works the same way. b = a on a list or dict makes b is a true, and editing through one edits the other. The rule holds for objects, lists, and dicts, not for plain numbers: a bare number is copied as a value, so b = a then b = 5 leaves a alone. Whether an assignment copies the value into a fresh place or just adds a second name to the old place is the copy versus share question, and it decides a large share of real bugs.

A name is not the thing. It is a label on a place. Two labels can sit on one place, and then changing the object through "either" name changes the one thing underneath.

4The function that changed your list

A function that "changed something it shouldn't have" is almost always this same thing, one step removed. Passing an argument is an assignment: the parameter becomes a second name for whatever you handed in. If you hand in a list, the function does not get a copy, it gets another name for your list.

# Python
def add_tag(tags):
    tags.append("new")   # edits the caller's list

my_tags = ["a", "b"]
add_tag(my_tags)
my_tags                  # ["a", "b", "new"] -- you never appended

Inside add_tag, the name tags points at the same list as my_tags outside. The append changes that one list, so the caller sees the extra item. Nothing here is broken; it is the shared-address rule from section 3, now crossing a function boundary.

The fix is to make a real copy before you change it, so the function works on its own place:

# Python: copy first
def add_tag(tags):
    tags = tags.copy()   # a fresh list, its own place
    tags.append("new")
    return tags

// JavaScript: same idea
let b = { ...a };        // a fresh object, not a shared address

A shallow copy like this is enough when the items themselves are not shared objects you also mutate; when they are, you need a deeper copy, which is where copy versus share picks up.

5Three questions when a line surprises you

When code does something you did not expect, pull the name, the place, and the value apart and ask three questions. Run them on the add_tag call above, where my_tags changed on its own:

What place does this name refer to? Inside the function, tags refers to the same list as my_tags, because passing the argument copied a shared address, not the list.

Did the line change the value in that place, or point a name at a different place? tags.append("new") changed the one list in place. It did not give tags a new list. So the change is visible everywhere that list is named, including back in the caller.

If instead the function had done tags = tags.copy() first, that assignment would have pointed tags at a fresh place, and the later append would touch only that copy. Same three parts, different answer: which is exactly how you can tell the two cases apart before running anything.

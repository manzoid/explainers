The stack and frames

When a function runs, its variables have to live somewhere: each call gets its own small chunk of memory, a frame. Once you can see frames stack up, two everyday puzzles clear up. Take a common one: a counter you keep in a local variable resets to zero on every call instead of counting up.

That reset, and the crash traceback you cannot make sense of, both come down to one well-organized part of memory: the part that holds a function's local variables while it runs and throws them away the instant it returns. (The memory page covered where values sit; this is one tidy region of that memory.) That part is a stack of frames. Once you see it, recursion stops being magic and a traceback turns into something you can read line by line.

1Every call needs its own scratch space

A function has parameters and local variables. Those are real values, and by the variables page every value sits at some address in memory. So when you call a function, the machine has to set aside a spot for that call's variables to live while it runs.

That spot is a frame (also called a stack frame or an activation record). One call, one frame. It holds three kinds of thing: the arguments passed in, the local variables the function declares, and a note saying where to return, the exact spot in the caller to resume once this call is done.

2Calls nest, so frames pile up

Functions call other functions. main calls cook, cook calls chop. While chop runs, cook is not finished; it is paused, waiting. Its frame has to stay put, holding its variables, until chop returns.

So the frames pile up. Each new call puts a fresh frame on top of the one that called it. The caller's frame sits underneath, frozen, until the call above it finishes and hands control back.

3The pile is a stack: last in, first out

The pile has a strict rule. The frame that went on last is the one that comes off first, because a function can only return after everything it called has returned. You never finish cook before chop; chop was last on, so it is first off. That last-in, first-out pile is the call stack.

The word "stack" is not a loose picture. It is the same last-in, first-out shape you get stacking plates: you add and remove only at the top. That simple rule is what lets the machine track a deep chain of calls with almost no bookkeeping.

4A local lives and dies with its frame

When a call returns, its frame comes off the stack. The memory it used is not carefully erased; it is simply handed back, free for the next call to overwrite.

A local variable lives exactly as long as its frame. It appears when the call starts and disappears when the call returns. That is why each fresh call starts with fresh locals, and why a value you want to keep has to be returned, not left behind in the frame.

You have probably hit this in Python without a name for it. Say you want a running count and try to keep it in a local:

def count_up():
    total = 0          # a brand-new local, rebuilt on every call
    total += 1
    return total

count_up()             # 1
count_up()             # 1 again, not 2

It resets to 0 every time, and now you can see exactly why: each call is a new frame, so total is a new local that starts from scratch. The frame from the last call, and its total, is already gone. To carry a value across calls you have to put it somewhere that outlives a single frame: return it and have the caller hold it, or store it on an object (a class attribute, say) that keeps living after the call returns.

In Python, JavaScript, Java, Go, and other managed languages that is the whole surprise: the runtime keeps a value alive as long as something you returned still refers to it. (In C you can even hand back the address of a dead frame and read freed memory, but Python simply will not let you write that bug.)

This is also the point where two ideas get blurred, so keep them apart. When a name stops working, it is easy to think the variable is "gone" because it left scope, when the real question is whether its frame is still live.

Lifetime is how long a local lives, the subject of this page. Visibility is who is allowed to see a name in the first place, a separate rule covered on the scope and globals page. A name can be out of scope while its frame is still alive, and a frame can be gone while the name is still in scope elsewhere.

5Recursion is just frames of the same function

Recursion looks mysterious until you see the stack. A function that calls itself is nothing special to the machine: each call gets its own frame, exactly like any other call.

def factorial(n):
    if n == 1:                 # base case
        return 1
    return n * factorial(n - 1)

factorial(3)                   # 3 frames stack up: (3), then (2), then (1)

Call factorial(3) and it calls factorial(2), which calls factorial(1). That is three frames stacked up, all running the same code, each with its own n. Because each frame has its own copy of the parameters, the n in one call does not disturb the n in another. They are different variables at different addresses that happen to share a name.

The frames unwind in reverse: factorial(1) returns 1 first, then factorial(2) returns 2, then factorial(3) returns 6, each handing its result down to the call that was waiting for it.

Try this right now. Put breakpoint() on the line just before return 1, then run the file. When it stops, the debugger's call stack shows all three factorial frames at once, each with its own value of n. (breakpoint() is the built-in way to do this from Python 3.7 on.)

6The stack has a size limit: overflow

The stack is a fixed region of memory, not endless. Every frame takes up some space, so a very deep chain of calls can fill it. When the machine tries to add one more frame past the limit, you get a stack overflow, and the program stops.

The usual cause is a recursion that never stops, a missing or wrong base case (the condition that stops the recursion), so it calls itself forever, piling on frames until the stack runs out. In Python this shows up as a RecursionError, raised once you pass its default limit of about 1000 nested calls; many other languages crash with a message that literally says stack overflow. Either way the cause is the same: the pile of frames hit the ceiling.

Very deep but correct recursion can overflow too. Some languages reuse the current frame for the last recursive call instead of adding a new one, so the depth stays flat; this is called tail-call optimization. CPython does not do it, so in Python deep recursion usually means rewriting it as a loop.

7Reading the stack

A crash traceback is the stack made visible. When a program dies, it prints the frames that were live at that instant, the exact chain of who called whom, from the outermost call down to the one that broke. Here is a short Python traceback:

Traceback (most recent call last):
  File "app.py", line 20, in <module>
    checkout(cart)
  File "app.py", line 12, in checkout
    total = apply_discount(price, count)
  File "app.py", line 6, in apply_discount
    return price / count
ZeroDivisionError: division by zero

Read it from the bottom. The last line is the error itself. Just above it is the frame where the crash happened: apply_discount, at line 6, dividing by a count that was zero. That is where the bug lives. Above that, each block is one paused frame, older as you go up: apply_discount was called by checkout at line 12, and checkout was called at line 20 at the top level. Python labels this "most recent call last", so the deepest, most recent frame is always at the bottom, right next to the error. Some languages print the same list in the opposite order, newest first; either way, the frame next to the error message is where things actually went wrong.

With the stack in view, a traceback stops being noise. It is just the pile of frames at the instant things broke, listed in order, and reading it top to bottom (or bottom to top) tells you exactly who called whom on the way to the bug. That is the payoff: the same picture that made recursion and vanishing locals clear also turns every future crash into a readable map.

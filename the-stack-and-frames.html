The stack and frames

When a function runs, its own working state has to live somewhere. Each call gets a frame: a small chunk of memory holding that call's parameters, its local variables, and where to go back when it finishes.

The state page said a running program holds information as it works, and the memory page said that information sits in numbered memory. This page is about one well-organized part of that memory: the part that answers "where do a function's locals live, and why are they gone the moment it returns?" Once you see it, recursion stops being magic and a whole class of bugs starts to make sense. It sits next to scope and globals, which is about who can see a variable; this page is about how long it lives.

1Every call needs its own scratch space

A function has parameters and local variables. Those are real values, and by the variables page every value sits at some address in memory. So when you call a function, the machine has to set aside a spot for that call's variables to live while it runs.

That spot is a frame (also called a stack frame or an activation record). One call, one frame. It holds three kinds of thing: the arguments passed in, the local variables the function declares, and a note saying where to return, the exact spot in the caller to resume once this call is done.

2Calls nest, so frames pile up

Functions call other functions. main calls cook, cook calls chop. While chop runs, cook is not finished; it is paused, waiting. Its frame has to stay put, holding its variables, until chop returns.

So the frames pile up. Each new call puts a fresh frame on top of the one that called it. The caller's frame sits underneath, frozen, until the call above it finishes and hands control back.

3The pile is a stack: last in, first out

The pile has a strict rule. The frame that went on last is the one that comes off first, because a function can only return after everything it called has returned. You never finish cook before chop; chop was last on, so it is first off. That last-in, first-out pile is the call stack.

The word "stack" is not a loose picture. It is the same last-in, first-out shape you get stacking plates: you add and remove only at the top. That simple rule is what lets the machine track a deep chain of calls with almost no bookkeeping.

4A local lives and dies with its frame

When a call returns, its frame comes off the stack. The memory it used is not carefully erased; it is simply handed back, free for the next call to overwrite. So every local variable in that frame is gone. Not hidden, not saved somewhere, gone.

A local variable lives exactly as long as its frame. It appears when the call starts and disappears when the call returns. That is why each fresh call starts with fresh locals, and why a value you want to keep has to be returned, not left behind in the frame.

In a language with raw pointers, like C or the unsafe parts of Swift, this bites in a sharp way: you can take the address of a local and hand it back, but the memory it points at is already freed the instant the call returns.

# C, roughly:
int* make() {
    int n = 42;      // n lives in make's frame
    return &n;       // hands back the address of n
}                    // frame gone; that address now points at freed memory

Python and other managed languages do not let you do this: you hand back the value, and the runtime keeps whatever it points at alive for you. So you cannot trip on a freed frame the way C does. But the lifetime rule is the same underneath, which is why the plain "the local does not persist to the next call" surprise shows up everywhere.

5Recursion is just frames of the same function

Recursion looks mysterious until you see the stack. A function that calls itself is nothing special to the machine: each call gets its own frame, exactly like any other call.

def factorial(n):
    if n == 1:                 # base case
        return 1
    return n * factorial(n - 1)

factorial(3)                   # 3 frames stack up: (3), then (2), then (1)

Call factorial(3) and it calls factorial(2), which calls factorial(1). That is three frames stacked up, all running the same code, each with its own n. Because each frame has its own copy of the parameters, the n in one call does not disturb the n in another. They are different variables at different addresses that happen to share a name.

The frames unwind in reverse: factorial(1) returns 1 first, then factorial(2) returns 2, then factorial(3) returns 6, each handing its result down to the call that was waiting for it. Run it under a debugger and pause inside factorial(1): the call stack shows all three frames at once, one n per frame.

6The stack has a size limit: overflow

The stack is a fixed region of memory, not endless. Every frame takes up some space, so a very deep chain of calls can fill it. When the machine tries to add one more frame past the limit, you get a stack overflow, and the program stops.

The usual cause is a recursion that never stops, a missing or wrong base case, so it calls itself forever, piling on frames until the stack runs out. In Python this shows up as a RecursionError, raised once you pass its default limit of about 1000 nested calls; many other languages crash with a message that literally says stack overflow. Either way the cause is the same: the pile of frames hit the ceiling. Very deep but correct recursion can overflow too, which is one reason some problems are rewritten as loops instead.

7Reading the stack

Keep the two ideas apart. The frame explains lifetime: how long a local lives, and why it vanishes the moment the call returns. Visibility, who is allowed to see a name in the first place, is a separate rule the language enforces, and that is the subject of scope and globals.

When you next see a crash trace, that list of function names is a snapshot of the stack at the moment things went wrong: the frames that were live, top to bottom, the exact chain of who-called-whom. Reading it is reading the state, which is what this whole series is about.

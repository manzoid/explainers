Memory: bytes at addresses

Memory is one long row of numbered cells. Each cell holds one byte, a small number. Each cell has a number of its own, its address. Everything a program holds lives here, as bytes in cells.

You have already met this row without knowing its name. Run id(x) in a Python notebook and you get back a plain integer. Ask numpy for an int32 array instead of int64 and it takes half the memory for the exact same numbers. Hit a bug and a debugger shows your variable as 0x7ffe... instead of its name, or a crash prints an address of 0x0. All four are the same fact showing through: your program's data lives in memory, and memory is nothing but numbered cells holding bytes.

Read this and those four things stop being trivia: you will be able to picture where your data actually sits and reason about it. That picture is what arrays, pointers, and the stack are all laid over, and where you drop back to when a program stops making sense.

1One long row of cells, each holding a byte

Picture memory as a single row of small boxes, side by side, running off into the distance. There are no named sections and no nested structure, just one flat line of identical cells. Every cell is the same size and holds the same kind of thing: one byte, a whole number from 0 to 255. Not a letter directly, not a word, not a decimal point. Just a number in that range. (A byte is eight bits, and eight on/off switches have 256 settings, counted 0 to 255.)

A real machine has billions of these cells, but the picture is the same whether there are ten or ten billion.

2Each cell has an address

The cells are numbered in order: 0, 1, 2, 3, and on up. A cell's number is its address. The address is fixed. It is where the cell sits in the row, not what it holds. The cell at address 4 is always the fifth box, whatever number is written inside it right now.

On a 64-bit machine there are billions of cells, so real addresses are huge numbers. That is why a debugger shows something like 0x7ffe1234abcd: it is just one cell's number, written in the short form covered in section 5.

Keep the two numbers apart: the address is which cell, the byte is what is in it. Both are numbers, and mixing them up is the source of a lot of confusion later.

3Read and write, both by address

The processor does two basic things with memory, and both start with an address. It can read: give it an address, and it hands back the byte in that cell. It can write: give it an address and a byte, and it puts that byte in that cell, overwriting whatever was there. Read the byte at 3. Write 200 to cell 5. That is the whole conversation. The processor never searches or scans for a value; it names the cell by number and goes straight there. In RAM, unlike a hash map or a database index, every cell is at a fixed physical position, so any cell is as quick to reach as any other.

That fixed position is the reason an interviewer expects you to say array indexing is O(1). When you write my_list[5], the code jumps straight to that cell; it does not count from the front. numpy makes the arithmetic plain: the values sit in a row starting at some base address, and the address of element i is just

address = base + i * element_size

one multiply and one add, no scan, whatever i is.

Indexing an array is O(1) because the cell number is a fixed calculation, not a search. That is why an interviewer treats "index into the array" as free while "search the array" is not.

One honest detail: in numpy and C-style arrays the cells hold the literal bytes of each value, so element i lands right on its value. A Python list stores pointers to objects instead, so the cell you land on holds an address, not the number itself. The same O(1) calculation still applies, but there is one more step to reach the value. Pointers are their own page.

You can even see one of these addresses from Python. In CPython, id(x) is the memory address of the object x names, which is why two variables with the same id() are the same object in the same cells, not two copies that happen to match. (The language only promises id() is unique per object; CPython happens to use the address.) Which cell a name in your code stands for is what variables, values, and addresses covers next.

4Bigger values span several cells

One byte only reaches 255, so anything larger will not fit in a single cell. The fix is simple: use several cells in a row and read them as one value. Four cells give 32 bits, which covers about four billion values; the machine treats those four adjacent bytes as one number.

How many cells a value gets is fixed by its type, and numpy is where you can watch this happen. A dtype is exactly this count: how many cells each element takes. An int32 gets 4 cells per number, an int64 gets 8. So a one-million-element int64 array occupies 8 million consecutive cells, and the same array as int32 occupies 4 million. Same values, half the memory. You can check it directly:

a = np.zeros(1_000_000, dtype=np.int64)
a.itemsize   # 8   cells per element
a.nbytes     # 8_000_000

b = a.astype(np.int32)
b.itemsize   # 4
b.nbytes     # 4_000_000

This is why a dtype mismatch quietly changes the size of a whole array, and why picking int32 over int64 is a real decision when memory is tight. The same rule holds in any language: the type fixes the cell count. In C, sizeof(int) is usually 4 and sizeof(double) is 8; in Java or C# the runtime fixes it for you, same effect.

Text works the same way. A character is stored as a number, using a standard character-to-number table. The oldest such table is ASCII, where each character maps to one byte: H is 72 and i is 105, exactly the two highlighted cells in the diagram in section 2. So the word Hi is two cells in a row, cell 2 holding 72 and cell 3 holding 105. You can see the numbers yourself with ord('H'), which returns 72.

Modern text can use more than one byte per character (Python strings pick 1, 2, or 4 bytes internally depending on the characters), but the idea does not change: a character is a number, a string is a run of adjacent cells, and its address is the address of its first cell.

5Why your tools show hex

A cell holding 72 does not know whether it is the letter H, the number 72, or one slice of a bigger number. The meaning is not in the cell; it lives in the code that decides how to read each region, the same point the state page made about bits. That is why a debugger does not try to show you a friendly value for a raw location. It shows you the address itself, and for addresses the natural format is hex.

Why hex and not plain decimal? Because each hex digit stands for exactly four bits, so two hex digits describe one byte cleanly, with nothing left over. One byte holds 0 to 255, and 255 is 0xFF in hex: two digits, one byte, done. The byte 72 is 0x48. Every byte maps to exactly two hex digits, no waste, no overlap.

Decimal does not line up like that. Take the address 32766. In hex it is 0x7FFE, and you can read the bytes straight off it: 0x7F is 127, the first byte, and 0xFE is 254, the second byte. In decimal, 32766 is five digits that do not line up with any byte boundary. That clean fit is the whole reason memory tools count in hex: the number on screen matches the underlying bytes.

So this row is the picture your tools drop into when things go wrong. A debugger prints a variable as 0x7ffe... instead of a name because that number is the address of the variable's first cell, not the value inside. A crash on a null pointer shows 0x0 because the program tried to read at cell zero, an address kept invalid on purpose, instead of at a real one. Once you know the number is an address, 0x0 tells you exactly what went wrong: you followed a reference that pointed nowhere.

You hit the same mistake in Python, just without the hex. Python's None means "no object here," so reading an attribute off it, AttributeError: 'NoneType' object has no attribute ..., is the Python version of reading from 0x0: you asked for something at a place that was never given a real value. Confuse the address with the byte and these read as noise; keep them apart and each one tells you what happened.

6Where this goes next

Every value your program holds ends up here: a count, a name, a price, a whole list, a picture, all of it bytes in numbered cells and nothing else. Higher up, a language gives you names and types and shapes so you rarely think about raw addresses, but underneath it is this row the entire time. When a value surprises you, if id(x) and id(y) come back equal, they share a cell, so changing one changes the other, they are the same object.

From here, variables, values, and addresses shows how a name in your code gets attached to a cell, then arrays and pointers are just two shapes laid over the row you now have.

Reuse and share freely.

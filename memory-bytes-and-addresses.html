Memory: bytes at addresses

Memory is one long row of numbered cells. Each cell holds one byte, a small number. Each cell has a number of its own, its address. Everything a program holds lives here, as bytes in cells.

You have already met this row without knowing its name. Run id(x) in a Python notebook and you get back a plain integer. Ask numpy for an int32 array instead of int64 and it takes half the memory for the exact same numbers. Hit a bug and a debugger shows your variable as 0x7ffe... instead of its name, or a crash prints an address of 0x0. All four are the same fact showing through: your program's data lives in memory, and memory is nothing but numbered cells holding bytes.

This page builds that model from the ground up, so those four things stop being trivia and start being one picture you can reason about. It is also what arrays, pointers, and the stack are all laid over, and where you drop back to when a program stops making sense, because down here there are only two questions: which cell, and what byte is in it. Section 5 turns that into the payoff you can use right away, why your tools speak in hex and what an address in a crash is telling you. The state page said a running program is always holding information that is real and sits somewhere. This page is that somewhere.

1One long row of cells, each holding a byte

Picture memory as a single row of small boxes, side by side, running off into the distance. There are no named sections and no nested structure, just one flat line of identical cells. Every cell is the same size and holds the same kind of thing: one byte, a whole number from 0 to 255. Not a letter directly, not a word, not a decimal point. Just a number in that range. (A byte is eight bits, and eight on/off switches have 256 settings, counted 0 to 255.)

A real machine has billions of these cells, but the picture is the same whether there are ten or ten billion.

2Each cell has an address

The cells are numbered in order: 0, 1, 2, 3, and on up. A cell's number is its address. The address is fixed. It is where the cell sits in the row, not what it holds. The cell at address 4 is always the fifth box, whatever number is written inside it right now.

Keep the two numbers apart: the address is which cell, the byte is what is in it. Both are numbers, and mixing them up is the source of a lot of confusion later.

3Read and write, both by address

The processor does two basic things with memory, and both start with an address. It can read: give it an address, and it hands back the byte in that cell. It can write: give it an address and a byte, and it puts that byte in that cell, overwriting whatever was there. Read the byte at 3. Write 200 to cell 5. That is the whole conversation. The processor never searches or scans for a value; it names the cell by number and goes straight there, so any cell is as quick to reach as any other.

This is the reason an interviewer expects you to say array indexing is O(1). When you write my_list[5], Python jumps straight to that cell; it does not count from the front. It takes the same time whether the list has ten items or ten million. numpy makes the arithmetic plain: the values sit in a row starting at some base address, and the address of element i is just

address = base + i * element_size

one multiply and one add, no scan, whatever i is. That single calculation is what O(1) means here, and it is why an interviewer treats "index into the array" as free while "search the array" is not.

You can see the address itself. In CPython, id(x) returns the memory address of the object x names, which is why two variables with the same id() are the same object in the same cells, not two copies that happen to match. Where these addresses come from, which cell a name in your code stands for, is what variables, values, and addresses covers next.

4Bigger values span several cells

One byte only reaches 255, so anything larger will not fit in a single cell. The fix is simple: use several cells in a row and read them as one value. Four cells give 32 bits, which covers about four billion values; the machine treats those four adjacent bytes as one number.

How many cells a value gets is fixed by its type, and numpy is where you can watch this happen. A dtype is exactly this count: how many cells each element takes. An int32 gets 4 cells per number, an int64 gets 8. So a one-million-element int64 array occupies 8 million consecutive cells, and the same array as int32 occupies 4 million. Same values, half the memory. You can check it directly:

a = np.zeros(1_000_000, dtype=np.int64)
a.itemsize   # 8   cells per element
a.nbytes     # 8_000_000

b = a.astype(np.int32)
b.itemsize   # 4
b.nbytes     # 4_000_000

This is why a dtype mismatch quietly changes the size of a whole array, and why picking int32 over int64 is a real decision when memory is tight. The same rule holds in any language: the type fixes the cell count. In C, sizeof(int) is usually 4 and sizeof(double) is 8; in Java or C# the runtime fixes it for you, same effect.

Text works the same way. Each character is stored as a number in one byte, so the word Hi is two cells: 72 for H, then 105 for i (the two highlighted cells in the diagram above). A longer word is a longer run of cells. The pattern is always the same: a big thing is a group of adjacent cells, and its address is the address of its first cell.

5Why your tools show hex

A cell holding 72 does not know whether it is the letter H, the number 72, or one slice of a bigger number. The meaning is not in the cell; it lives in the code that decides how to read each region, the same point the state page made about bits.

So why do addresses show up as hex like 0x7ffe... and not plain decimal? Because each hex digit stands for exactly four bits, so two hex digits describe one byte cleanly, with nothing left over. That neat fit, one byte per pair of digits, is the whole reason memory tools count in hex: the number on screen lines up with the underlying bytes instead of smearing across them the way a decimal number would.

This row is the picture your tools drop into when things go wrong. A debugger prints a variable as 0x7ffe... instead of a name because that number is the address of the variable's first cell, not the value inside. A crash on a null pointer shows an address of 0x0 because the program tried to read at cell zero, an address kept invalid on purpose, instead of at a real one. You hit the same mistake in Python without the hex: following a None reference and getting AttributeError: 'NoneType' object has no attribute ... is the same error one level up, you told the program to go somewhere that holds nothing useful. Confuse the address with the byte and these read as noise; keep them apart and each one tells you exactly what happened.

6Everything is bytes in cells

Every value your program holds ends up here. A count, a name, a price, a whole list, a picture: all of it is bytes in numbered cells, and nothing else. Higher up, a language gives you names and types and shapes so you rarely think about raw addresses, but underneath, it is this row of cells the entire time.

Next comes how a name in your code gets attached to a cell in this row, in variables, values, and addresses. After that, an array is just values packed into cells in a row, and a pointer is a cell whose content is itself an address, the number of some other cell. Both are shapes laid over the row you now have.

The four things that opened this page, the integer from id(x), the array that shrank when you changed its dtype, the 0x7ffe... in a debugger, the 0x0 in a crash, are one row of numbered cells seen through different tools. When a value surprises you, changes when you did not expect it, crashes, or takes more space than it should, drop back here and ask the two questions: which cell, and what byte is in it.

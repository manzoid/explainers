Memory: bytes at addresses

Memory is one long row of numbered cells. Each cell holds one byte, a small number. Each cell has a number of its own, its address. Everything a program holds lives here, as bytes in cells.

The state page said a running program is always holding some information, and that the information is real and sits somewhere. This page is that somewhere. It is the ground the rest of the series stands on: arrays, pointers, and the stack are all just patterns laid over these numbered cells. Get this picture solid and the rest stops being mysterious.

1One long row of cells

Picture memory as a single row of small boxes, side by side, running off into the distance. There is no shape to it beyond that: no named sections, no nested structure, just one flat line of identical cells.

Every cell is the same size and holds the same kind of thing: one byte. A real machine has billions of these cells, but the idea is the same whether there are ten or ten billion.

2Each cell holds one byte

A byte is a number from 0 to 255. That is all a single cell can hold: one whole number in that range. Not a letter directly, not a word, not a decimal point as you usually write one. Just a number, 0 through 255.

Why 255? A byte is eight bits, eight on/off switches, and eight switches have 256 different settings, which we count as 0 to 255. You do not have to think in bits to use this page. Take the cell as a box that holds one number from 0 to 255, and that is enough.

3Each cell has an address

The cells are numbered in order: 0, 1, 2, 3, and on up. A cell's number is its address. The address is fixed. It is where the cell sits in the row, not what it holds. The cell at address 4 is always the fifth box, whatever number is written inside it right now.

Keep the two numbers apart: the address is which cell, the byte is what is in it. Both are numbers, and mixing them up is the source of a lot of confusion later.

4The CPU works by address

The processor does two basic things with memory, and both start with an address. It can read: give it an address, and it hands back the byte in that cell. It can write: give it an address and a byte, and it puts that byte in that cell, overwriting whatever was there.

That is the whole conversation with memory. Read the byte at 3. Write 200 to cell 5. The processor never searches or scans for a value; it names the cell it wants by number and goes straight there. Any cell is as quick to reach as any other, because reaching it is just naming its address. That is how the model works; in practice the processor keeps recently-used cells close and is slower on the rest, but that is a later detail.

Where those addresses come from, which cell a name in your code stands for, is what variables, values, and addresses is about, and that is a later page.

5Bigger values span several cells

One byte only reaches 255, so anything larger will not fit in a single cell. The fix is simple: use several cells in a row and read them as one value. Four cells give you 32 bits, which covers about four billion different values, so a number up to about four billion takes four cells side by side; the machine treats those four adjacent bytes as one number. The program decides how many cells belong together, which in C is what the type does: it fixes whether a value gets 1 cell or 4.

The order those bytes are stored in matters too, and different processors handle it differently; that detail comes up when you look at raw bytes, not before.

Text works the same way. Each character is stored as a number in one byte, so the word Hi is two cells: 72 for H, then 105 for i, the two highlighted cells in the diagram above. A longer word is a longer run of cells. The pattern is always the same: a big thing is a group of adjacent cells, and its address is the address of its first cell.

6The cells do not know what they mean

A cell holding 72 does not know if it is the letter H, the number 72, part of a larger number, or one piece of something else. It is just a byte. The meaning is not in the cell; it lives in the program, in the code that decides how to read each region.

This is the same point the state page made: bits are just a pattern until a program reads them a certain way. The same run of cells could be a number to one part of the program and text to another. Nothing in memory settles it. The code does.

7Everything is bytes in cells

Every value your program holds ends up here. A count, a name, a price, a whole list, a picture: all of it is bytes in numbered cells, and nothing else. Higher up, a language gives you names and types and shapes so you rarely think about raw addresses, but underneath, it is this row of cells the entire time.

So whenever a program stops making sense, you can drop back to this row and ask just two questions: which cell, and what byte is in it. An array is only values packed into cells in a row; a pointer is a value stored across several cells, and that value is itself an address, the number of some other cell. Different shapes, same ground: bytes at addresses.

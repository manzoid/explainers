Aliasing bugs

Aliasing is when two or more names point at the same piece of state, so changing it through one name changes it for all of them. Take a common case: a and b both hold the same list, you append to it through a, and now b has changed too, though you never touched b. Forget the sharing is there and you get some of the most confusing bugs in everyday programming.

A variable that holds a list or a dict does not hold the list itself. It holds a reference, a pointer to one object sitting in memory. Assign that variable to a second name and you copy the pointer, not the object, so both names now reach the same list. Change it through one, and the other sees the change.

The examples use Python, which shares by default and makes the trap easy to hit, but the same bug lives in JavaScript, Java, and Go, any language where a variable can hold a reference. This is also a common source of wrong output on live coding problems: you mutate the input array in place, or state leaks from one test case into the next, and your function passes on the first example and fails on the second. Interviewers know these traps and watch for them. Five bugs follow, each with its fix, then one habit that catches all of them. (The pointer and copy vs share pages explain the underlying model.)

1The surprise, in one picture

Two names, a and b, both point at one list. You change the list through a. Then you read it through b and find it changed too, because there was never a second list to begin with.

Every bug below is a variation on this same picture. Carry one habit into all of them, the two questions in the box, and none of them can catch you.

Two questions catch almost every case.Is the object mutable, something you can change in place, like a list, dict, or set? And will more than one name write to it? If both are yes, copy it, or make sure only one name ever writes. A number or a string is not mutable, so sharing one is always safe.

2Two variables, one object: the base case

Start with the plainest form, just assignment. You point one variable at another and think you now have two independent values. You have two names for one value.

defaults = {"theme": "dark", "font": "sans"}
settings = defaults           # not a copy, a second name

settings["theme"] = "light"
# defaults["theme"] is now "light" as well

Assignment copies the pointer, not the object. This is safe until someone mutates the shared object, which is why it can hide for a long time: everything works until the day someone writes through one of the names. Then a change over here shows up over there, far from where you were looking. The fix is the same everywhere on this page, so learn it once: make a real copy when you mean a separate thing, or make sure only one name ever writes to the shared object. Here: settings = dict(defaults), or a deep copy if it nests.

The same trap sits in JavaScript, Go, and Java: assigning an array or object gives you a second name, not a second copy. In JavaScript it is one line to see for yourself:

JavaScript

const a = [1, 2, 3];
const b = a;                  // second name, not a copy
b.push(9);
console.log(a);               // [1, 2, 3, 9] — a changed too

3Changing a collection while looping over it

This one can pass your tests and still fail in production. You loop over a list and remove items as you go. On some inputs it drops the right items; on others it silently skips one, and which inputs fail depends on where the matches sit. If your test cases happen to put the matches apart, everything looks fine.

nums = [1, 2, 4, 5]
for n in nums:
    if n % 2 == 0:
        nums.remove(n)        # editing the list we are walking
# nums is [1, 4, 5]: 4 is even, so it should be gone, but it survived

The loop tracks its position by a hidden index. Walk [1, 2, 4, 5] step by step:

position 0 -> 1   odd, keep, move on
position 1 -> 2   even, remove it. now [1, 4, 5], and 4 slides into position 1
position 2 -> 5   the loop skipped position 1, so it never sees 4

Removing 2 shifted everything after it left by one, so 4 dropped into a slot the loop already passed. Whenever two items to remove sit next to each other, the second one slips through.

Fix: loop over one thing and build a separate result, so the thing you read and the thing you change are not the same object.

nums = [n for n in nums if n % 2 != 0]   # new list, old one untouched

The exact index-shift is a Python detail; JavaScript's for...of does not renumber the same way. But the broader rule holds in every language: do not read from one thing and write to it at the same time. Read from one, write to another.

4Appending to a list you thought was private

You pass a list into a function or store it on an object, expecting the function to have its own copy. It does not. It has the same list, and when it appends, your list grows.

def build_tags(base):
    base.append("draft")      # meant to work on a local copy
    return base

user_tags = ["python", "web"]
build_tags(user_tags)
# user_tags is now ["python", "web", "draft"]

Passing a list to a function does not copy the list. It copies the pointer, so the parameter base and your user_tags are two names for one list. A name being local to the function does not make the object it points at local. This is the same shape as section 2, just with the second name created by a function call instead of a plain assignment.

Fix: make a real copy at the boundary, right where the list comes in.

def build_tags(base):
    base = list(base)         # fresh list, same contents
    base.append("draft")
    return base               # caller's list is untouched

5The shared mutable default argument

This one is Python-specific. It catches almost everyone once. You give a function a list parameter with a default of the empty list, so callers can leave it out:

def add_item(item, bucket=[]):     # looks harmless
    bucket.append(item)
    return bucket

add_item("a")   # ['a']
add_item("b")   # ['a', 'b']  ... where did 'a' come from?

The default value [] is built once, when the function is defined, not once per call. So every call that leaves out bucket shares that same one list. The hidden second name here is one you never wrote: the language made it for you by reusing the default. Because that shared state survives between calls, it is a classic way for one test to poison the next.

Java, C#, JavaScript, and Go have no mutable default arguments, so this exact bug cannot happen there. It stays on the list because it is the purest example of the underlying trap: the language hands you a second name to one object, and you never see it get made.

Fix: do not share in the first place. Use None as the default and make a fresh list inside the call.

def add_item(item, bucket=None):
    if bucket is None:
        bucket = []            # a new list every call
    bucket.append(item)
    return bucket

6The shallow copy that still shares

So you learned to copy. But a shallow copy duplicates only the top-level container; any containers inside it are still shared. If your dict holds a list, the copy gets its own dict but points at the same inner list.

original = {"name": "cart", "items": ["apple"]}
clone = dict(original)         # a shallow copy

clone["items"].append("pear")
# original["items"] is now ["apple", "pear"] too

dict(original) gives you a new outer dictionary, but its "items" key still holds the same list the original does. You copied the box and left the inner box shared. The copy worked for the top level, so you stop worrying, and the sharing hides one layer down.

Fix: copy deeply when the contents are themselves mutable. Python has a tool for exactly this.

import copy
clone = copy.deepcopy(original)   # inner list is copied too

A deep copy costs more, so reach for it only when you know the contents are shared and mutable. Often the cheaper answer is to treat the data as read-only and never mutate it, building a new object instead.

In front-end code (React)That read-only habit is the whole idea behind immutable updates. Instead of changing the old state object, you spread it into a new one and override the key you want: 
const next = { ...state, count: state.count + 1 };
Nothing else holding the old state gets surprised, because you never wrote through the shared object.
In data work (pandas)Slicing a DataFrame can hand back a view onto the original, not a copy, so writing through it can change the source: 
sub = df[df["price"] > 100]   # may be a view
sub["price"] = 0              # may also zero df, or warn, depending on the slice
That ambiguity is what SettingWithCopyWarning is telling you about. When you mean a separate frame, say so: sub = df[df["price"] > 100].copy().

7How to spot it, even in someone else's code

You do not have to memorize five bugs. You need one habit: when a mutable object gets a second name, ask whether both names should see the writes. In a code review, second names are easy to catch once you know the shapes. Each is a plain assignment or pass that looks like a copy and is not:

  • Assignment (b = a). The plain assign makes a second name, not a second object. Fix: b = list(a) or dict(a).
  • Passing into a function (f(a)). The parameter is a second name for your object. Fix: copy at the top of the function, base = list(base).
  • Returning and keeping (return self.items). The caller walks away holding the same object the function held. Fix: return list(self.items), or agree to treat it as read-only.
  • Storing on an object (self.items = a). The field becomes a second name for the caller's list. Fix: self.items = list(a).
  • A default list or dict argument (def f(x, acc=[]), Python). The default is one shared object. Fix: default to None, build a fresh one inside.
  • A shallow copy of nested data (dict(a), a[:]). Copies the top layer, shares the inner ones. Fix: copy.deepcopy(a) when the contents are mutable.

Every one is the picture from section 1: a second name for one object. When you see it, run the two questions from the box in section 1, and you are done.

The point

A variable holds a reference, not the object. So the whole family of bugs comes down to one habit: when a mutable object picks up a second name, decide whether both names should see the writes. If not, copy it, or let only one name write. Two questions, asked every time, and the confusing bugs stop being confusing.

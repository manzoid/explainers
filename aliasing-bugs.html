Aliasing bugs

When two or more names point at the same piece of state, changing it through one name changes it for all of them. That sharing is called aliasing, and when you forget it is there, you get some of the most confusing bugs in everyday programming.

A variable that holds a list or a dict does not hold the list itself. It holds a reference, a pointer to one object sitting in memory. Assign that variable to a second name and you copy the pointer, not the object, so both names now reach the same list. Change it through one, and the other sees the change. The fix is almost always one of two things: make a real copy, or make sure only one name ever writes to the shared object.

The examples use Python, which shares by default and makes the trap easy to hit, but the same bug lives in JavaScript, Java, and Go, any language where a variable can hold a reference. Five bugs follow, each with its fix, then one habit that catches all of them. (The pointer and copy vs share pages explain the underlying model.)

1The surprise, in one picture

Two names, a and b, both point at one list. You change the list through a. Then you read it through b and find it changed too, because there was never a second list to begin with.

Every bug on this page is this picture wearing a disguise. Carry one habit into all of them, the two questions in the box below, and the disguises stop working.

Two questions catch almost every case.Is the object mutable, something you can change in place, like a list, dict, or set? And will more than one name write to it? If both are yes, copy it, or make sure only one name ever writes. A number or a string is not mutable, so sharing one is always safe.

2Two variables, one object: the base case

Start with the plainest form, just assignment. You point one variable at another and think you now have two independent values. You have two names for one value.

defaults = {"theme": "dark", "font": "sans"}
settings = defaults           # not a copy, a second name

settings["theme"] = "light"
# defaults["theme"] is now "light" as well

Assignment copies the pointer, not the object. This is safe until someone mutates the shared object, which is why it can hide for a long time: everything works until the day someone writes through one of the names. Then a change over here shows up over there, far from where you were looking. The same trap sits in JavaScript, Go, and Java: assigning an array or map gives you a second name, not a second copy. (Swift is the exception. Its arrays and dictionaries copy on assignment, so this particular trap does not reach them.)

Fix: copy when you mean a separate thing. settings = dict(defaults), or a deep copy if it nests.

3Changing a collection while looping over it

This one can pass your tests and still fail in production. You loop over a list and remove items as you go. On some inputs it drops the right items; on others it silently skips one, and which inputs fail depends on where the matches sit. If your test cases happen to put the matches apart, everything looks fine.

nums = [1, 2, 4, 5]
for n in nums:
    if n % 2 == 0:
        nums.remove(n)        # editing the list we are walking
# nums is [1, 4, 5]: 4 is even, so it should be gone, but it survived

The loop tracks its position by index. It reads index 1 (2) and removes it. Everything after shifts left by one, so 4 slides down into index 1, a slot the loop already passed. The loop steps on to index 2 and never looks back at 4. Whenever two items to remove sit next to each other, the second one slips through.

Fix: loop over one thing and build a separate result, so the thing you read and the thing you change are not the same object.

nums = [n for n in nums if n % 2 != 0]   # new list, old one untouched

4Appending to a list you thought was private

You pass a list into a function or store it on an object, expecting the function to have its own copy. It does not. It has the same list, and when it appends, your list grows.

def build_tags(base):
    base.append("draft")      # meant to work on a local copy
    return base

user_tags = ["python", "web"]
build_tags(user_tags)
# user_tags is now ["python", "web", "draft"]

Passing a list to a function does not copy the list. It copies the pointer, so the parameter base and your user_tags are two names for one list. A name being local to the function does not make the object it points at local.

Fix: make a real copy at the boundary, right where the list comes in.

def build_tags(base):
    base = list(base)         # fresh list, same contents
    base.append("draft")
    return base               # caller's list is untouched

5The shared mutable default argument

This one is Python-specific. It catches almost everyone once. You give a function a list parameter with a default of the empty list, so callers can leave it out:

def add_item(item, bucket=[]):     # looks harmless
    bucket.append(item)
    return bucket

add_item("a")   # ['a']
add_item("b")   # ['a', 'b']  ... where did 'a' come from?

The default value [] is built once, when the function is defined, not once per call. So every call that leaves out bucket shares that same one list. The disguise here is that you never wrote a second name for the list; the language did, quietly, by reusing the default.

Java, C#, JavaScript, and Go have no mutable default arguments, so this exact bug cannot happen there. It stays on the list because it is the purest example of the underlying trap: the language hands you a second name to one object, and you never see it get made.

Fix: do not share in the first place. Use None as the default and make a fresh list inside the call.

def add_item(item, bucket=None):
    if bucket is None:
        bucket = []            # a new list every call
    bucket.append(item)
    return bucket

6The shallow copy that still shares

So you learned to copy. But a plain copy of a container copies only the top layer. If the container holds other containers, the copy still points at the same inner ones.

original = {"name": "cart", "items": ["apple"]}
clone = dict(original)         # a shallow copy

clone["items"].append("pear")
# original["items"] is now ["apple", "pear"] too

dict(original) gives you a new outer dictionary, but its "items" key still holds the same list the original does. You copied the box and left the inner box shared. The disguise is that the copy worked for the top level, so you stop worrying, and the sharing hides one layer down.

Fix: copy deeply when the contents are themselves mutable. Python has a tool for exactly this.

import copy
clone = copy.deepcopy(original)   # inner list is copied too

A deep copy costs more, so reach for it only when you know the contents are shared and mutable. Often the cheaper answer is to not share at all, or to treat the data as read-only and never mutate it. That read-only habit is the whole idea behind immutable updates in front-end code: in a React reducer you write {...state, key: val} to build a new top-level object instead of changing the old one, so nothing else holding the old object gets surprised.

Data folks meet the same thing as view versus copy: df['col'] on a pandas DataFrame can hand back a view onto the original, so mutating it sometimes changes the source and sometimes does not, depending on how you sliced. When in doubt, take an explicit .copy().

7How to spot it, even in someone else's code

You do not have to memorize five bugs. You need one habit: when a mutable object gets a second name, ask whether both names should see the writes. In a code review, second names are easy to catch once you know the shapes. Each is a plain assignment or pass that looks like a copy and is not:

  • Assignment (b = a). The plain assign makes a second name, not a second object. Fix: b = list(a) or dict(a).
  • Passing into a function (f(a)). The parameter is a second name for your object. Fix: copy at the top of the function, base = list(base).
  • Returning and keeping (return self.items). The caller walks away holding the same object the function held. Fix: return list(self.items), or agree to treat it as read-only.
  • Storing on an object (self.items = a). The field becomes a second name for the caller's list. Fix: self.items = list(a).
  • A default list or dict argument (def f(x, acc=[]), Python). The default is one shared object. Fix: default to None, build a fresh one inside.

Every one is the picture from section 1: a second name for one object. When you see it, run the two questions. Is the object mutable, and will more than one name write to it? If both are yes, copy it, or let only one name write. The same reference thinking explains the failures on the null and dangling pointers page, a name that points at nothing, or at something already gone.

Reuse and share freely.

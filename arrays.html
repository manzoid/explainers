Arrays

An array is the plainest way to lay out a row of values in memory, and that plain layout is what makes it fast enough to sit under so many coding-interview problems. Here is the payoff: look up the millionth element and the computer does not scan through a million values to reach it. It does one multiply and one add to jump straight to the spot, the same tiny bit of work it would spend on the first element.

The state page said a program holds information somewhere. The memory page said that somewhere is a long line of numbered bytes, each with its own address, and that one sentence is all you need here. An array is the first real shape you can give that line: take some of those bytes and treat them as a row of slots that all hold the same kind of value. Below the Python lists and JavaScript arrays you use every day sits this plainer thing, an array of raw values, and it is the one worth understanding first.

1A run of equal-size cells

Memory is a line of bytes, each with an address. An array claims a stretch of that line and cuts it into cells of one fixed size: say four bytes each for a row of 32-bit integers. Cell 0 comes first, then cell 1 right after it, then cell 2, with no gaps. The cells sit next to each other in address order. That property has a name: the array is contiguous.

The array knows two things about itself: where it starts, called the base address, and how big one cell is. It does not store a separate address for every element. It does not need to, and that is the whole trick.

2Finding element i is just arithmetic

Since the cells are equal in size and packed with no gaps, the address of any element is a plain calculation. Element i sits i cells past the base, and each cell is size bytes, so:

address of element i = base + i * size

Say the base is address 1000 and each cell is 4 bytes. Element 0 is at 1000, element 1 at 1004, element 3 at 1000 + 3 * 4 = 1012. When you write a[3] in Python, JavaScript, or almost any language, this is the arithmetic that runs underneath: one multiply, one add, then a read at that address.

The work does not grow with the size of the array. One multiply, one add, one read, whether you want the first element or the ten-millionth. That is what people mean when they call array indexing O(1), or constant time. O(1) is Big-O notation, a shorthand for how the work grows as the input grows; O(1) means it does not grow at all, so the cost stays flat whether the array holds ten values or ten million.

Compare that to a linked list, where the elements are scattered around memory and each one only holds a pointer to the next. There is no formula for the millionth element. To reach it you follow a million pointers, one after another. The array skips all of that by paying up front: keep every cell the same size and packed together, and any position is just math.

3The Python list is not this

Here is where a lot of working programmers carry a fuzzy picture. A Python list or a JavaScript array can hold mixed things: a number, a string, another list, all in one row. Those objects are different sizes, so they cannot be packed into equal cells. The base + i * size formula would fall apart.

What sits in the row instead is a set of equal-size references: each slot is a pointer to the real object, which lives somewhere else in memory. So the row itself is a plain array, an array of pointers, and indexing it is still O(1). You get the pointer instantly, then follow it to the object. That convenience, holding mixed types and growing as you go, costs you one extra hop per access and leaves the objects spread around memory rather than packed together.

A Python list is a plain array of pointers. The array of raw values is the simpler idea sitting underneath it.

4Packed cells are also fast to scan

Instant indexing is the headline, but the packed layout pays off again when you loop over the whole array in order, which is most of what real code does. When the CPU reads one value, it also pulls the neighboring values into the cache, a small pool of very fast memory sitting next to the processor. Reading from cache is roughly 100 times faster than reaching all the way out to main memory, and those neighbors are exactly the elements the loop reaches next. Packed values stream through the cache well; objects scattered around memory do not.

This is why a NumPy array iterates much faster than a Python list of the same numbers: the NumPy array is raw values packed back to back, while the list is pointers to integer objects scattered around memory. (Java shows the same split between an int[] and an ArrayList<Integer>.) In an interview this is the fact that decides a data-structure choice: when you just need to walk a row of numbers and speed matters, reach for the packed array of raw values, not the list of objects.

5Reading past the end

The formula base + i * size computes an address for any i you hand it, even one past the end of the array. Ask for element 10 of a six-element array and it still produces an address, one that points at memory the array does not own. What happens next depends entirely on the language.

JavaScript, one of your two daily languages, is the loose one, and this is the real footgun: a read past the end quietly returns undefined instead of complaining.

In JavaScript, arr[arr.length] returns undefined, not an error. Nothing flags the bad line; the undefined flows into the next line and the failure surfaces far away, which is what makes off-by-one bugs there so slippery.

Python is stricter: it raises an IndexError the moment you go out of range (Java and C# throw similar exceptions), so the mistake stops right there with a message that points at the line. In lower-level languages like C there is no check at all, so the read simply happens and you get garbage or a crash, a classic source of security bugs. You will not hit that in Python or JavaScript, but the off-by-one behind it is the same, and it is the most common surprise beginners meet with arrays.

6Why this is the shape interviews push around

Most array interview questions come down to one thing: moving indices through a fixed row, reading or swapping values as you go. The array is the only basic structure where an index, a fixed order, and constant-time access all line up at once, and the standard patterns all lean on that combination.

Take reversing an array in place. Put one index at the left and one at the right, swap those two cells, then step the left index forward and the right index back, and stop when they meet. Here it is in Python, short enough to run in your head on [1, 2, 3, 4], which comes out [4, 3, 2, 1]:

def reverse(a):
    left, right = 0, len(a) - 1
    while left < right:
        a[left], a[right] = a[right], a[left]
        left += 1
        right -= 1

That is the two-pointer pattern: keep two indices and move them by a rule, usually inward from both ends. No new memory, no scanning, just index arithmetic on a fixed row. The same shape checks whether "racecar" reads the same both ways: compare the outer pair, step inward, stop when they meet. Reach for two pointers when the answer depends on a pair of positions. Three more patterns are worth knowing by name before a screen. Each is its own topic, so treat this as a preview, a set of labels to hang your reading on, not the full lesson:

  • Sliding window. Keep a start and an end index that mark off a stretch of the array, and slide them along to find the best stretch that fits some rule. (A substring is one such stretch of a string: the characters from one position to another, in order.) For "longest substring with no repeated character," on "abcabcbb" the window grows to "abc", then slides forward each time a repeat appears; the answer is length 3. Reach for it on "longest or shortest run of..." questions.
  • Prefix sums. Walk the array once building a running total, so the sum of any slice becomes a single subtraction. For [3,1,4,1,5] the running totals are [3,4,8,9,14], and elements 1 through 3 sum to 9 - 3 = 6. Reach for it when a question asks about the sums of many different ranges in a large array.
  • Binary search. This one needs a sorted array: jump to the middle, compare, throw away the half that cannot hold the answer, and repeat. To find 7 in [1,3,5,7,9], check the middle (5), then search only the right half. It rides on O(1) indexing, the ability to jump to the middle for free, more than on the packed layout itself, which is why it usually gets its own page. Reach for it when the array is sorted and you need a value or a boundary.

Once you see the array as a fixed row of numbered cells, the machinery behind all of these is the same: O(1) indexing lets you jump anywhere and move on.

That is why the array comes first. It is the plainest shape state can take, and the one the most common problems are built on. The next page, an array problem in memory, takes the reversal above and works it out cell by cell.

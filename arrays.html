Arrays

Look up the millionth element of an array and the computer does not scan through a million values to find it. It does one multiply and one add to compute the exact spot, then reads it directly. That is the same tiny bit of work it would spend on the first element. The speed comes from one rule: an array is a row of equal-size cells, packed back to back in memory.

The state page said a program holds information somewhere. The memory page said that somewhere is a long line of numbered bytes, each with its own address; if you have not read it, that one sentence is all you need here. An array is the first real shape you can give that line: take some of those bytes and treat them as a row of slots that all hold the same kind of value. This page is about the plain array of raw values, the idea underneath the fancier lists you use every day. The next one, an array problem in memory, puts it to work.

1A run of equal-size cells

Memory is a line of bytes, each with an address. An array claims a stretch of that line and cuts it into cells of one fixed size: say four bytes each for a row of 32-bit integers. Cell 0 comes first, then cell 1 right after it, then cell 2, with no gaps. The cells sit next to each other in address order. That property has a name: the array is contiguous.

The array knows two things about itself: where it starts, called the base address, and how big one cell is. It does not store a separate address for every element. It does not need to, and that is the whole trick.

2Finding element i is just arithmetic

Since the cells are equal in size and packed with no gaps, the address of any element is a plain calculation. Element i sits i cells past the base, and each cell is size bytes, so:

address of element i = base + i * size

Say the base is address 1000 and each cell is 4 bytes. Element 0 is at 1000, element 1 at 1004, element 3 at 1000 + 3 * 4 = 1012. When you write a[3] in Python, JavaScript, or almost any language, this is the arithmetic that runs underneath: one multiply, one add, then a read at that address.

The work does not grow with the size of the array. One multiply, one add, one read, whether you want the first element or the ten-millionth. That is what people mean when they call array indexing O(1), or constant time: the cost stays flat as the array grows. Compare that to a linked list, where the elements are scattered around memory and each one only holds a pointer to the next. There is no formula for the millionth element; to reach it you follow a million pointers, one after another. The array skips all of that by paying up front: keep every cell the same size and packed together, and any position is just math.

3Reading past the end

The formula gives a valid address only when i is between 0 and length minus 1. Ask for element 10 of a six-element array and the formula still computes an address, one that points at memory outside the array. In the languages you write, this is mostly handled for you, so it turns into a clear error rather than a silent disaster. But the two languages you use most handle it differently, and the difference matters.

Python raises an IndexError the moment you go out of range (Java and C# raise similar exceptions), so the mistake stops right there with a message that points at the line. JavaScript is looser, and this is the real footgun for you: a read past the end quietly returns undefined instead of throwing. Write arr[arr.length] by mistake and nothing complains; the undefined flows into the next line, and the failure surfaces far from where the off-by-one actually happened.

In lower-level languages like C there is no check at all, so the read simply happens and you get garbage or a crash, which is a classic source of security bugs. You will not hit that in Python or JavaScript, but the off-by-one behind it is still a real bug, and it is the most common surprise beginners meet with arrays.

4The Python list is not this

Here is where a lot of working programmers carry a fuzzy picture. A Python list or a JavaScript array can hold mixed things: a number, a string, another list, all in one row. Those objects are different sizes, so they cannot be packed into equal cells. The base + i * size formula would fall apart.

What sits in the row instead is a set of equal-size references: each slot is a pointer to the real object, which lives somewhere else in memory. So the row itself is a plain array, an array of pointers, and indexing it is still O(1). You get the pointer instantly, then follow it to the object. The convenience costs you that extra hop and the objects being spread around.

A Python list is a plain array of pointers. The array of raw values is the simpler idea sitting underneath it.

5Packed cells are also fast to scan

Instant indexing is the headline, but the packed layout pays off again when you loop over the whole array in order, which is most of what real code does. When the CPU reads one value, it also pulls the neighboring values into a small pool of very fast memory, and those neighbors are exactly the elements the loop reaches next. Packed values stream through that fast memory well; objects scattered around memory do not.

This is why a NumPy array iterates much faster than a Python list of the same numbers: the NumPy array is raw values packed back to back, while the list is pointers to integer objects scattered around memory. (The same split shows up in Java between an int[] and an ArrayList<Integer>.) For an interview you do not need the hardware detail; the point to carry is small: scanning a packed array is faster than chasing scattered pointers, and that is why arrays are the default when you just need to walk a row of numbers.

6Why this is the shape interviews push around

The array is the only basic structure where three things line up at once: every element has an index, the elements keep a fixed order, and any one of them is reachable in constant time. Most array interview questions are really about exploiting that combination, moving indices through the row and swapping values as you go.

Take reversing an array in place. Put one index at the left and one at the right, swap those two cells, then step the left index forward and the right index back, and stop when they meet. Here it is in Python, short enough to run in your head on [1, 2, 3, 4], which comes out [4, 3, 2, 1]:

def reverse(a):
    left, right = 0, len(a) - 1
    while left < right:
        a[left], a[right] = a[right], a[left]
        left += 1
        right -= 1

No new memory, no scanning, just index arithmetic on a fixed row. That same shape drives a large share of array questions. A few patterns are worth knowing by name before a screen, each with a small example to picture:

  • Two pointers. Keep two indices and move them by a rule, usually inward from both ends. Reversing [1,2,3,4] into [4,3,2,1] is one; checking that "racecar" reads the same both ways is another (compare the outer pair, step inward, stop when they meet). Reach for it when the answer depends on a pair of positions.
  • Sliding window. Keep a start and an end index that mark off a stretch of the array, and slide them along to find the best stretch that fits some rule. For "longest substring with no repeated character," on "abcabcbb" the window grows to "abc", then slides forward each time a repeat appears; the answer is length 3. Reach for it on "longest or shortest run of..." questions.
  • Prefix sums. Walk the array once building a running total. For [3,1,4,1,5] the running totals are [3,4,8,9,14]. Now the sum of any slice is one subtraction: elements 1 through 3 sum to 9 - 3 = 6. Reach for it when a question asks about the sums of many different ranges in a large array.
  • Binary search. On a sorted array, jump to the middle, compare, throw away the half that cannot hold the answer, and repeat. To find 7 in [1,3,5,7,9], check the middle (5), then search only the right half. Reach for it when the array is sorted and you need a value or a boundary.

Once you see the array as a fixed row of numbered cells, the machinery behind all of these is the same: O(1) indexing lets you jump anywhere and move on.

That is why the array comes first. It is the plainest shape state can take, and the one the most common problems are built on. The next page, an array problem in memory, takes the reversal above and works it out cell by cell.

Arrays

Ask an array for its millionth element and it does not go looking. It does one multiply and one add, lands on the exact byte, and hands it back, the same tiny bit of work it would spend on the first element. That trick comes from one rule: an array is a row of equal-size cells, packed back to back in memory.

The state page said a program holds information somewhere, and the memory page said that somewhere is a long line of numbered bytes. An array is the first real shape you can give that line: take some of those bytes and treat them as a row of slots that all hold the same kind of value. This page is about the plain C-style array, the idea underneath the fancier lists you use every day. The next one, an array problem in memory, puts it to work.

1A run of equal-size cells

Memory is a line of bytes, each with an address. An array claims a stretch of that line and cuts it into cells of one fixed size: say four bytes each for a row of 32-bit integers. Cell 0 comes first, then cell 1 right after it, then cell 2, with no gaps. The cells sit next to each other in address order. That property has a name: the array is contiguous.

The array knows two things about itself: where it starts, called the base address, and how big one cell is. It does not store a separate address for every element. It does not need to, and that is the whole trick.

2Finding element i is just arithmetic

Since the cells are equal in size and packed with no gaps, the address of any element is a plain calculation. Element i sits i cells past the base, and each cell is size bytes, so:

address of element i = base + i * size

Say the base is address 1000 and each cell is 4 bytes. Element 0 is at 1000, element 1 at 1004, element 3 at 1000 + 3 * 4 = 1012. To read element 3 the machine does one multiply and one add, then goes to that address.

The work does not grow with the size of the array. One multiply, one add, one read, whether you want the first element or the ten-millionth. That is what people mean when they call array indexing O(1), or constant time: the cost stays flat as the array grows. Compare that to a linked list, where the elements are scattered around memory and each one only holds a pointer to the next. To reach the millionth you would follow a million pointers, one after another. The array skips all of it by paying up front: keep every cell the same size and packed together, and any position is just math.

One catch: the formula gives a valid address only when i is between 0 and length minus 1. Go outside that range and you read memory that is not yours. In C that is undefined behavior, often a silent wrong value or a crash; in a managed language like Python or Java you get an exception instead. This out-of-bounds read is the most common surprise beginners hit with arrays.

3The Python list is not this

Here is where a lot of working programmers carry a fuzzy picture. A Python list or a JavaScript array can hold mixed things: a number, a string, another list, all in one row. Those objects are different sizes, so they cannot be packed into equal cells. The base + i * size formula would fall apart.

What sits in the row instead is a set of equal-size references: each slot is a pointer to the real object, which lives somewhere else in memory. So the row itself is a plain array, an array of pointers, and indexing it is still O(1). You get the pointer instantly, then follow it to the object. The convenience costs you that extra hop and the objects being spread around.

A Python list is a plain array of pointers. The C-style array of raw values is the simpler idea sitting underneath it.

4Packed cells are also fast to scan

Instant indexing is the headline, but the same layout pays off again when you walk the whole array in order, which is most of what real code does.

The reason is how memory reaches the CPU. A modern CPU does not fetch one byte at a time. It grabs a fixed chunk, usually 64 bytes, called a cache line, and copies it into fast on-chip storage. So a read at element 3 also pulls elements 4 through 7 along with it, for free. In a packed array the neighbors are the next elements you were going to read, so one trip to memory serves several loop steps.

There is a second gift. The CPU notices a straight-line walk through memory and starts fetching the next chunk before your code asks for it. This is the prefetcher, and on a packed array it means the data you need next is often already on-chip by the time you get there. A linked list defeats both effects: each node sits at some unrelated address, so every step is a fresh trip to slow memory with nothing prefetched.

Those trips are not a small tax. Reading from on-chip L1 cache costs a few CPU cycles; reaching out to main memory (DRAM) costs on the order of a hundred times more. That gap is why iterating a NumPy array runs so much faster than iterating a Python list of the very same numbers: the NumPy array is raw values packed back to back, streaming through cache, while the list is pointers to integer objects scattered across the heap, one slow miss after another.

5Why this is the shape interviews push around

The array is the only basic structure where three things line up at once: every element has an index, the elements keep a fixed order, and any one of them is reachable in constant time. Most array interview questions are really about exploiting that combination, moving indices through the row and swapping values as you go.

Take reversing an array in place. Put one index at the left and one at the right, swap those two cells, then step the left index forward and the right index back, and stop when they meet:

while left < right: swap a[left], a[right]; left = left + 1; right = right - 1

No new memory, no scanning, just index arithmetic on a fixed row. Many classic problems are variations on that same idea of walking indices inward from both ends or sliding a window along. Once you see the array as a fixed row of numbered cells, the machinery is the same each time.

That is why the array comes first. It is the plainest shape state can take, and the one the most common problems are built on. The next page, an array problem in memory, takes one such problem and works it out cell by cell.

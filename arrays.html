Arrays

Ask an array for its millionth element and it does not go looking. It does one multiply and one add, lands on the exact byte, and hands it back, the same tiny bit of work it would spend on the first element. That trick comes from one rule: an array is a row of equal-size cells, packed back to back in memory.

The state page said a program holds information somewhere. The memory page said that somewhere is a long line of numbered bytes, each with its own address; if you have not read it, that one sentence is all you need here. An array is the first real shape you can give that line: take some of those bytes and treat them as a row of slots that all hold the same kind of value. This page is about the plain C-style array, the idea underneath the fancier lists you use every day. The next one, an array problem in memory, puts it to work.

1A run of equal-size cells

Memory is a line of bytes, each with an address. An array claims a stretch of that line and cuts it into cells of one fixed size: say four bytes each for a row of 32-bit integers. Cell 0 comes first, then cell 1 right after it, then cell 2, with no gaps. The cells sit next to each other in address order. That property has a name: the array is contiguous.

The array knows two things about itself: where it starts, called the base address, and how big one cell is. It does not store a separate address for every element. It does not need to, and that is the whole trick.

2Finding element i is just arithmetic

Since the cells are equal in size and packed with no gaps, the address of any element is a plain calculation. Element i sits i cells past the base, and each cell is size bytes, so:

address of element i = base + i * size

Say the base is address 1000 and each cell is 4 bytes. Element 0 is at 1000, element 1 at 1004, element 3 at 1000 + 3 * 4 = 1012. When you write a[3] in Python, JavaScript, C, or almost any language, this is the arithmetic that runs underneath: one multiply, one add, then a read at that address.

The work does not grow with the size of the array. One multiply, one add, one read, whether you want the first element or the ten-millionth. That is what people mean when they call array indexing O(1), or constant time: the cost stays flat as the array grows. Compare that to a linked list, where the elements are scattered around memory and each one only holds a pointer to the next. There is no formula for the millionth element; to reach it you follow a million pointers, one after another. The array skips all of that by paying up front: keep every cell the same size and packed together, and any position is just math.

3Reading past the end

The formula gives a valid address only when i is between 0 and length minus 1. Ask for element 10 of a six-element array and it happily computes an address anyway, one that points at memory outside the array. What happens next depends entirely on your language, and the difference is large.

In a managed language, the runtime checks the index for you. Python, Java, and C# raise an exception the moment you go out of range, so the mistake stops right there with a clear message. Plain JavaScript arrays are looser: a read past the end quietly gives back undefined instead of throwing, which can let the bug travel a long way before it shows.

In C there is no check at all. The read just happens, and you get whatever bytes were sitting at that address: a garbage value, or a crash, or nothing wrong until much later. This is undefined behavior, and reads and writes past the end of an array are one of the classic sources of security holes. Most people reading this write in a managed language, so the usual cost is an exception, not a breach. Either way, the out-of-bounds slip is the most common surprise beginners hit with arrays.

4The Python list is not this

Here is where a lot of working programmers carry a fuzzy picture. A Python list or a JavaScript array can hold mixed things: a number, a string, another list, all in one row. Those objects are different sizes, so they cannot be packed into equal cells. The base + i * size formula would fall apart.

What sits in the row instead is a set of equal-size references: each slot is a pointer to the real object, which lives somewhere else in memory. So the row itself is a plain array, an array of pointers, and indexing it is still O(1). You get the pointer instantly, then follow it to the object. The convenience costs you that extra hop and the objects being spread around.

A Python list is a plain array of pointers. The C-style array of raw values is the simpler idea sitting underneath it.

5Packed cells are also fast to scan

Instant indexing is the headline, but the same layout pays off again when you walk the whole array in order, which is most of what real code does. A modern CPU does not fetch one byte at a time. It grabs a chunk of about 64 bytes at once, called a cache line, into fast on-chip storage. In a packed array, reading one element pulls its neighbors along for free, and those neighbors are exactly the elements your loop reaches next. The CPU also notices a straight walk through memory and starts fetching the next chunk before you ask for it. The word for a layout that plays this well with the hardware is cache-friendly.

This is why a NumPy array iterates far faster than a Python list of the same numbers: the NumPy array is raw values packed back to back, streaming through cache, while the list is pointers to integer objects scattered around memory. The same split shows up in Java: an int[] packs the numbers themselves, while an ArrayList<Integer> holds pointers to boxed Integer objects sitting apart. Packed beats scattered, and by a wide margin on a long array.

6Why this is the shape interviews push around

The array is the only basic structure where three things line up at once: every element has an index, the elements keep a fixed order, and any one of them is reachable in constant time. Most array interview questions are really about exploiting that combination, moving indices through the row and swapping values as you go.

Take reversing an array in place. Put one index at the left and one at the right, swap those two cells, then step the left index forward and the right index back, and stop when they meet. Here it is in Python, short enough to run in your head on a four-element array:

def reverse(a):
    left, right = 0, len(a) - 1
    while left < right:
        a[left], a[right] = a[right], a[left]
        left += 1
        right -= 1

No new memory, no scanning, just index arithmetic on a fixed row. That same shape drives a large share of array questions. A few patterns are worth knowing by name before a screen:

  • Two pointers. Keep two indices and move them by a rule, either inward from both ends (reversing, or checking a palindrome) or both forward at different speeds. Reach for it when the answer depends on a pair of positions.
  • Sliding window. Keep a start and an end index that mark off a stretch of the array, and slide them along to find the best stretch that fits some rule. "Window" just means that moving stretch. Reach for it on "longest or shortest run of..." questions.
  • Prefix sums. Walk the array once building a running total, so afterward the sum of any slice is a single subtraction. Reach for it when a question asks about sums of many ranges.

Binary search belongs in the same family when the array is sorted: it uses O(1) indexing to jump to the middle, throw away half, and repeat. Once you see the array as a fixed row of numbered cells, the machinery behind all of these is the same.

That is why the array comes first. It is the plainest shape state can take, and the one the most common problems are built on. The next page, an array problem in memory, takes one such problem and works it out cell by cell.

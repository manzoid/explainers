Scope and globals

A value in your program is wrong. You search for where it gets set and find four places, in three files, any of which could have run. Nothing tells you which one is at fault. That confusion has a cause and a name: scope, and what happens when you throw it away.

Every name in a program has a region of code where it can be used; outside that region it does not exist. That region is its scope. Scope is how you decide who is allowed to see and change a piece of state. The state page said the trouble in programs comes from state that changes; the stack and frames page showed where a function keeps its own state while it runs. This page is about the next question: once you have some state, who is allowed to touch it?

1A local name has a short list of suspects

Write total inside a function and the rest of the program cannot see it. The name lives in that function's frame, is born when the call starts, and is gone when the call returns. Only code inside that function can read or write it. So when total comes out wrong, the list of suspects is short: it is the code in that function, and nowhere else.

One surprise for Python programmers: Python has no block scope. A variable made inside a loop or an if does not disappear when the block ends. It lives until the whole function returns.

for i in range(3):
    pass
print(i)      # 2  -- i is still here after the loop

This bites when you reuse i later and get a leftover value, or read a loop variable after a loop that never ran and hit a NameError. (JavaScript's let and const do have block scope; the old var does not, which is its own source of surprises.) The boundary that holds everywhere is the function: that is the box that keeps a name to itself.

2A global is visible everywhere

At the outermost level is the global scope: a module-level name, sitting in no function. Because it is in no box, every box can reach it. Any function, anywhere in the program, can read it, and any function can write it.

That looks convenient: define it once, reach it from anywhere, no passing it around. But the diagram already shows the cost. When config holds a wrong value, both setup and reset could be to blame, and nothing in the code tells you which one ran, or when. The convenience is up front; the confusion comes later.

3The cost: hidden coupling and order

State is manageable only when you know who can change it. A local name has a short list of suspects. A global has no list: when its value is wrong, the code that set it could be anywhere, run on any path, at any time.

Two functions that both touch the same global are joined together, even though nothing in their code says so. One writes it, the other reads it, and now they depend on each other through a channel you cannot see by reading either one. Worse, they depend on order. Say setup() fills in a global that process() reads:

config = {}

def setup():
    config["ready"] = True

def process():
    return work(config["ready"])   # KeyError if setup() has not run

Now the two must run in that order, but nothing in the code says so. Call process() first, or add a caller a month later that forgets setup(), and you get a stale value or a crash on a path that used to work, with no change nearby to explain it.

Scope contains state. It limits the set of code that can touch a value, so when the value is wrong you know where to look. A global throws that limit away, and takes the answer with it.

4Tests start fighting each other

Here is where this stops being abstract. A good test sets up a small world, runs one thing, and checks the result. A global breaks that, because it is one shared value that outlives any single test.

users = []              # module-level global

def test_add():
    users.append("alice")
    assert len(users) == 1

def test_empty():
    assert users == []   # green on its own, red after test_add runs

Run the whole file and one test leaves users dirty for the next. Now a test passes or fails depending on which ran first. Run the file and it is green; run one test alone, or reorder them, and it is red. The test did not change. The shared value it inherited did.

If you work in notebooks, you have already met this. Every cell shares one module-level namespace, so a cell that sets a global partway through a session quietly changes what later cells see. Re-run cells out of order and the results shift, with no line of code to blame.

5What to do instead

Not every global is bad. True constants and configuration you load once at startup and never touch again are fine, because nobody writes them during the run. The trouble is the global that gets written mid-run, by more than one place. For those, two habits fix it, and they are close to the same idea.

Pass state in explicitly. If a function needs a value, take it as an argument and return what you produce. Then the link between caller and function is written down in the call itself.

Before: the function reaches out to a shared global.

total = 0

def add(n):
    global total
    total = total + n   # who else writes total? anyone, anywhere

After: the function takes what it needs and hands back the result.

def add(total, n):
    return total + n     # the caller owns total; the trail is right here

Same arithmetic, no shared state. You can read who supplies the value and who receives it, and a wrong result has a short trail to follow. (One caution when you pass an object instead of a plain number: two functions can end up sharing one thing under two names. That is a separate, narrower problem, covered on the aliasing page.)

Keep every name in the narrowest scope that works. Declare it inside the loop if the loop is all that uses it; inside the function if only that function uses it; widen it only when a real need forces you to, and no further. The smaller the scope, the shorter the list of code that can be at fault, which is the whole point: the next time a value comes out wrong, you want a short list, not the whole program.

Reuse and share freely.

Scope and globals

Every name in a program has a region of code where it can be used, and outside that region it does not exist. That region is its scope. Scope is how you decide who is allowed to see and change a piece of state.

The state page said the trouble in programs comes from state that changes. The stack and frames page showed where a function keeps its own state while it runs. This page is about a related question: once you have a piece of state, who is allowed to touch it? Scope is the answer, and a global is what you get when you refuse to answer it.

1Scope is where a name is usable

Write total inside a function and the rest of the program cannot see it. The name lives in that function's frame, is born when the call starts, and is gone when the call returns. That region, the block of code where the name means something, is its scope.

Scope is a rule the language enforces at the level of names, not bits. The bits are just memory. Scope decides which parts of the code are even allowed to write a name that points at them.

2Scopes nest, and inner can see outer

Scopes sit inside each other like boxes. A function has an outer scope; a loop or an if inside it has a smaller scope of its own. Code in an inner box can reach the names of the boxes around it, but code outside cannot reach in. In most modern statically-typed languages a name declared deep inside a loop does not exist once the loop ends. (Some languages are looser here: Python has no block scope, so a loop variable outlives its loop, and older JavaScript with var did the same.)

So most names have a short life and a small audience. That is the point of them: a total local to one function is state that only that function can touch, which means only that function's code can be at fault when it goes wrong.

3A global is visible everywhere

At the outermost level is the global scope. A name here is not inside any box, so every box can see it. Any function, anywhere in the program, can read it, and any function can write it.

That sounds convenient. You define it once and reach it from anywhere, no passing it around. The cost is hidden, and it shows up later.

4The cost: you cannot tell who changed it

State is only manageable when you know who can change it. A local name has a short list of suspects: the code in its box, and nowhere else. A global has no list. When its value is wrong, the code that set it could be anywhere, run at any time, on any path.

Scope is how you contain state. It limits the set of code that can touch a value, so when the value is wrong, you know where to look. A global throws that limit away.

5Hidden coupling and order-dependence

Two functions that both touch the same global are joined together, even if nothing in their code says so. One writes it, the other reads it, and now they depend on each other through a channel you cannot see by reading either one. That is hidden coupling: a connection with no line drawn between the parts.

It gets worse with order. If setup() has to run before process() because it fills in a global that process() reads, then the two must run in that order, and nothing forces it. Call them the other way, or add a third caller later, and it breaks in a way that looks like nothing changed. (This is the exact problem when you run notebook cells out of order: a later cell reads a value that an earlier cell was supposed to set, and if you skipped that cell, you get a stale value or an error.) This is order-dependence: correctness that leans on an unwritten schedule.

6Tests fight each other

A good test sets up a small world, runs one thing, and checks the result. A global breaks that, because it is one shared value that outlives any single test. One test writes it, the next test reads what the first left behind, and now the second passes or fails depending on which ran first.

So tests that should be independent start interfering, and you get failures that appear and vanish depending on order. This is the same order-dependence from the last section, showing up where it hurts most: the code you were trusting to catch bugs.

Put the three failure modes together, cannot tell who changed it, hidden coupling, tests that fight, and you have why the phrase "globals considered harmful" stuck. The harm is not the keyword. It is the loss of containment: a global is scope removed, everything can reach in, and uncontained changing state is exactly the thing this whole series is about.

7What to do instead

Two habits, and they are the same habit seen twice.

Pass state in explicitly. If a function needs a value, give it as an argument and return what it produces. Now the connection between caller and function is written down in the call itself. You can read who supplies the value and who receives it, and a wrong value has a short trail to follow.

Keep every name in the narrowest scope that works. Declare it inside the loop if the loop is all that uses it; inside the function if only that function uses it. Widen the scope only when a real need forces you to, and no further. The smaller the scope, the shorter the list of code that can be at fault.

Passing values in has a close cousin to watch for: once you hand the same object to two places, both can change it through their own name, a problem called aliasing. That is its own trap, covered on the aliasing page.

Reuse and share freely.

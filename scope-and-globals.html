Scope and globals

A name in your program can only be changed from certain places, and that region is its scope; lose track of it and a wrong value has too many suspects. Take a common case: a value is wrong, you go looking for where it gets set, and find four places across three files, any of which could have run, with nothing to say which one is at fault.

Once you have some state, this page answers one question about it: who is allowed to touch it? Get that right and a wrong value has a short list of suspects. Get it wrong, with a global, and the list is the whole program.

1A local name has a short list of suspects

Write total inside a function and the rest of the program cannot see it. The name lives in that function's frame, is born when the call starts, and is gone when the call returns. Only code inside that function can read or write it. So when total comes out wrong, the list of suspects is short: it is the code in that function, and nowhere else.

One surprise for Python programmers: Python has no block scope. A variable made inside a loop or an if does not disappear when the block ends. It lives until the whole function returns.

for i in range(3):
    pass
print(i)      # 2  -- i survives the loop and holds its last value

This bites two ways. You reuse i later and silently get a leftover value. Or the loop never runs, so the name was never created, and reading it crashes:

for i in range(0):    # range(0) is empty, so the body never runs
    pass
print(i)              # NameError: name 'i' is not defined

New Python programmers rarely expect that one, because an empty loop looks harmless. The boundary that holds everywhere is the function: that is the box that keeps a name to itself. (JavaScript's let and const do have block scope; the old var does not, which is its own source of surprises.)

2A global is visible everywhere

At the outermost level is the global scope: a module-level name, sitting in no function. Because it is in no box, every box can reach it. Any function, anywhere in the program, can read it, and any function can write it.

That looks convenient: define it once, reach it from anywhere, no passing it around. The convenience is up front; the confusion comes later, and the next two sections are what that confusion looks like in code you have actually written.

3The cost: order-dependent bugs

If you work in notebooks, you have already met this. Every cell shares one module-level namespace, so a cell that sets a global partway through a session quietly changes what every later cell sees. Re-run cells out of order, or run them top to bottom in a fresh kernel, and the numbers shift, with no line of code changed to blame. The state you are reading was left there by a cell you ran ten minutes ago.

The same thing wrecks tests, and here it is concrete. A good test builds a small world, runs one thing, and checks the result. A shared global breaks that, because it outlives any single test:

users = []              # module-level global

def test_add():
    users.append("alice")
    assert len(users) == 1

def test_empty():
    assert users == []   # green on its own, red after test_add runs

Run the whole file and test_add leaves users dirty for test_empty. Now a test passes or fails depending on which ran first. Run one test alone and it is green; run the file, or reorder them, and it is red. The test did not change. The shared value it inherited did.

That is the deeper cost. Two functions that both touch one global are joined together, even though nothing in their code says so, and they depend on order: one must write before the other reads. Nothing writes that ordering down, so a caller added a month later that skips the write gets a stale value or a crash, on a path that used to work.

Scope contains state. It limits the set of code that can touch a value, so when the value is wrong you know where to look. A global throws that limit away, and takes the answer with it.

4What to do instead

Not every global is bad. Constants you set once at startup and never touch again are fine, because nobody writes them during the run:

MAX_RETRIES = 3
BASE_URL = "https://api.example.com"

These add no suspects: their value on line one is their value at the end. The trouble is the global that gets written mid-run, by more than one place. For those, two habits fix it.

Habit 1: pass state in explicitly.

If a function needs a value, take it as an argument and return what you produce. Then the link between caller and function is written down in the call itself.

Before

total = 0

def add(n):
    global total
    total = total + n   # who else writes total? anyone, anywhere

After

def add(total, n):
    return total + n     # the caller owns total; the trail is right here

Same arithmetic, no shared state. You can read who supplies the value and who receives it, and a wrong result has a short trail to follow. Python's global keyword, in the "before" version, lets a function write a module-level name, so you might wonder if it is the fix. It is not: it welds the function to a global it could have taken as an argument, and makes the tangle bigger, not smaller. (One caution when you pass an object instead of a plain number: two functions can end up sharing one thing under two names, a separate and narrower problem covered on the aliasing page.)

Habit 2: keep every name in the narrowest scope that works.

Declare a name inside the function if only that function uses it; inside the loop if only the loop uses it. Widen the scope only when a real need forces you to, and no further. A name that starts at module level, out of habit, is a global you did not have to create:

Before

seen = set()             # module level, but only summarize() uses it

def summarize(rows):
    for r in rows:
        seen.add(r.id)
    return len(seen)     # a second call finds seen already full

After

def summarize(rows):
    seen = set()         # born and dies inside the call
    for r in rows:
        seen.add(r.id)
    return len(seen)     # every call starts clean

Moving one line inward turns a shared global into a local. The second call now works, and the list of code that can touch seen shrinks from the whole module to four lines.

Interviewers probe this on purpose. A "what is wrong with this code" round often plants a mutable global and waits to see if you name it; a system design round asks what happens when two requests touch the same shared state at once. Knowing why the local version is safer is part of what a passing answer sounds like.

Do both habits and the opening scenario cannot happen. The value that came out wrong has one owner and a short trail, not four writers across three files. Who is allowed to touch it? Exactly the code you handed it to, and no one else.

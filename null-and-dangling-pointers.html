Null and dangling pointers

A pointer is supposed to point at something. Two ways it can let you down: it points at nothing, or it points at something that is gone. Follow either one and your program falls over.

The pointer page showed the basic idea: a pointer is a value that holds the address of some other data, so you can reach that data by following the pointer. This page is about the two most common ways that goes wrong. One of them, null, you meet every day in every language. The other, dangling, is mostly a concern in C and C++; if you work in Python, Java, Go, or JavaScript, you can read that part as background. If you have hit a NullPointerException, a segfault, or a "NoneType has no attribute" error, you have already met the first one.

1Following a pointer, and what "dereference" means

To dereference a pointer is to follow it and use whatever it points at: read the value there, or write a new one. That is the whole reason a pointer exists. Every crash on this page happens at the moment you dereference, not the moment the pointer was made.

So a bad pointer can sit quietly in your program for a long time. The trouble starts the instant something tries to follow it.

2Null: a pointer that points at nothing

Sometimes you have a pointer variable but no thing for it to point at yet. Maybe the data has not been loaded, or the lookup found no match. Languages give you a special value for "points at nothing on purpose": null in Java and C, nil in Go and Ruby, None in Python, nullptr in C++. Different name, same idea.

Null is fine to hold and pass around. The problem is dereferencing it. When you follow a pointer that points at nothing, there is no data to read, so the program stops:

  • Java throws a NullPointerException.
  • Python raises AttributeError: 'NoneType' object has no attribute ....
  • C or C++ usually gives a segfault, short for segmentation fault: the CPU refused an access to an address the program is not allowed to touch.

These are all the same error: you tried to use a value that was not there. Here it is in Python. The lookup can come back empty, and the next line follows it anyway:

# find_user can return None when nobody matches
user = find_user(user_id)
print(user.name)   # AttributeError if user is None

The fix is not clever. Check whether the value is there before you follow it, and decide what to do when it is not:

user = find_user(user_id)
if user is not None:
    print(user.name)
else:
    print("unknown")   # the empty case, handled on purpose

Every language has the same shape. Here is the same guard in Go, where the empty value is nil:

// Go
user := findUser(userID)
if user != nil {
    fmt.Println(user.Name)
} else {
    fmt.Println("unknown")
}

3Why null exists, and why it is a landmine

Null is handy. You often need a way to say "no value here yet" or "nothing matched", and a single shared null does the job in every pointer type. That is why it spread into so many languages.

The cost is that the type system usually cannot tell apart a pointer that has a real thing from one that is null. A variable that says User might hold a user, or might hold null, and both look the same until you follow it. So the check is on you, and it is easy to forget one place among hundreds.

Tony Hoare, who put null into a language in 1965, later called it his "billion-dollar mistake": one small convenience that has caused countless crashes and security holes across every codebase since.

This is why null is one of the highest-frequency crashes in real software. It is not exotic. It is the ordinary case of a value that might be missing, met by code that assumed it was there.

4Dangling: a pointer to something that is gone

If you work in Python, Java, Go, JavaScript, or C#, your garbage collector handles this for you, so it is mostly not your problem. It is a live concern mainly in C and C++. The rest of this section explains what actually happens under the hood.

First, why most readers can relax. Java, Python, Go, JavaScript, and C# use a garbage collector: memory is not freed while any pointer still points at it, so a dangling pointer mostly cannot happen. Rust checks lifetimes at compile time, so a dangling pointer will not even build. For most working programmers today, null is the daily problem and dangling is the one you meet only in C, in a native extension, or in a systems job.

Here is what a garbage collector is saving you from. A dangling pointer is not pointing at nothing, the way null does. It points at a real address, but the data that used to live there is gone. The pointer still holds the old address, but whoever lived there is gone. It shows up in two ways.

Freed memory. You handed the memory back with free or delete, but a pointer still holds that address. Following it now is the classic "use-after-free" bug.

// C
int *p = malloc(sizeof(int));
free(p);           // memory handed back
printf("%d", *p);  // use-after-free: p still holds the old address

Out of scope. You kept a pointer to a local variable after its stack frame returned. The slot that held the variable gets reused by the next call.

// C
int *make() {
    int x = 42;
    return &x;     // returns the address of a local; its slot is freed on return
}

Following a dangling pointer is worse than null in one way: it might not crash. The old bytes may still be sitting there, or they may already hold someone else's data. So you sometimes read a stale value, or quietly corrupt unrelated data, and the real symptom shows up far away. When two names share one piece of memory and one of them frees it, you get exactly the tangle covered in aliasing bugs.

5Habits that help you avoid these bugs

No amount of cleverness avoids these bugs. You avoid them with a few plain habits. The first three apply to almost every language. The last one is only for manual-memory code.

1. Check before you use. If a value might be null, test for it before you follow it. One check in the right place turns a crash into an ordinary branch you control.

# Python
if user is not None:
    send_email(user.address)

2. Prefer optionals over bare null. Many languages give you a type that says out loud "this might be empty": Optional in Java, Option in Rust, ? nullable types in Kotlin and Swift, and in Python the Optional annotation from typing. Now the reader of your code, and often the type checker, can see that the value might be missing, so it is harder to forget. In Python you write the possibility into the signature:

# Python
from typing import Optional

def find_user(user_id: int) -> Optional[User]:
    ...   # return a User, or None if nobody matches

In Java the Optional type lets you handle the empty case in one chain, with no if in sight, and no way to forget it:

// Java
String name = Optional.ofNullable(user)
    .map(User::getName)
    .orElse("unknown");

Kotlin and Swift build the same idea into the language. The type carries a ?, and short operators handle the empty case with a fallback value:

// Kotlin
val name = user?.name ?: "unknown"

// Swift
let name = user?.name ?? "unknown"

3. Fail fast with a clear message. If a value should never be null at some point, check it right there and stop with a message that names what went wrong, rather than letting a null travel three functions deep and blow up somewhere confusing. A crash near the cause is a gift; a crash far from the cause costs you an afternoon.

# Python
config = load_config(path)
assert config is not None, f"no config found at {path}"
# or raise explicitly:
if config is None:
    raise ValueError(f"no config found at {path}")

4. Null out and null-check freed pointers (C / C++ manual-memory only). After you free a pointer, set it to null right away, so a later accidental use fails loudly as a null instead of silently reading dead memory. If you write Python, Java, Go, or C#, skip this one; the garbage collector already covers it.

6The short version

A null pointer points at nothing; a dangling pointer points at something that is gone. Both are fine to hold and only crash when you dereference them.

So carry one working assumption. Any pointer that could be empty is empty until you have checked, and where the language offers it, let the type system carry that doubt for you. Do that and one of the most frequent crashes in real code becomes an ordinary case you handle on purpose.

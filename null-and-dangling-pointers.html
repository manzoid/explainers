Null and dangling pointers

A pointer is supposed to point at something. Two ways it can let you down: it points at nothing, or it points at something that is gone. Both are fine to hold and pass around. They only crash when you follow them.

The pointer page showed the basic idea: a pointer is a value that holds the address of some other data, so you can reach that data by following the pointer. This page is about the two most common ways that goes wrong. The first, null, you meet every day in every language. The second, dangling, you can mostly not trigger in Python or JavaScript, but knowing it explains a whole class of C and C++ crash reports you may read, and interviewers ask about it to see whether you understand what memory management is doing under you.

1Null: a pointer that points at nothing

Start with a bug you have hit. A lookup comes back empty, and the next line uses the result anyway:

# find_user can return None when nobody matches
user = find_user(user_id)
print(user.name)   # AttributeError: 'NoneType' object has no attribute 'name'

Reading user.name means "follow user to the object it points at, then read the name field." Following a pointer to use what it points at has a name: you dereference it. Here user points at nothing, so there is no object and no field to read, and Python stops.

"Points at nothing on purpose" is a value the language gives you. Maybe the data has not loaded yet, or the lookup found no match. It is called None in Python, null in Java and C, nil in Go and Ruby, nullptr in C++. Different name, same idea. Holding it is fine. Dereferencing it is the crash, and every language has its own word for the same failure:

  • Java throws a NullPointerException.
  • JavaScript says TypeError: Cannot read properties of null.
  • C or C++ usually gives a segfault, short for segmentation fault: the CPU refused an access to an address the program is not allowed to touch.

The fix is not clever. Check whether the value is there before you follow it, and decide what to do when it is not:

user = find_user(user_id)
if user is not None:
    print(user.name)
else:
    print("unknown")   # the empty case, handled on purpose

2Why null is a landmine

If the fix is one if, why is null one of the highest-frequency crashes in real software? Because the type system usually cannot tell a real value from null. A variable annotated User might hold a user, or might hold None, and both look the same until you follow one. Nothing forces you to check, so the check is on you, and it is easy to miss one place among the hundreds where the value flows through your code.

Tony Hoare, who put null into a language in 1965, later called it his "billion-dollar mistake": one small convenience that has caused countless crashes and security holes across every codebase since.

Null is not exotic. It is the ordinary case of a value that might be missing, met by code that assumed it was there. That is exactly why interviewers probe it: they want to see that you reach for the missing case on your own, before the crash finds it for you.

3Dangling: a pointer to something that is gone

If you work in Python, Java, Go, JavaScript, or C#, your garbage collector will not free memory while any pointer still points at it, so this bug is mostly not yours to hit. It is a live concern in C and C++. Read this as background: it is what the garbage collector is quietly saving you from.

A dangling pointer is not pointing at nothing, the way null does. It points at a real address, but the data that used to live there is gone. The classic case is use-after-free: you handed the memory back with free, but a pointer still holds that address, and following it now reads whatever moved in.

// C
int *p = malloc(sizeof(int));  // p now holds address 0x1a4c
free(p);           // memory handed back; p still holds 0x1a4c
printf("%d", *p);  // use-after-free: reads whatever lives at 0x1a4c now

The same thing happens if you return the address of a local variable: its stack frame goes away when the function returns, and the pointer is left aimed at a slot the next call will reuse.

Following a dangling pointer is worse than null in one way: it might not crash. The old bytes may still be sitting there, or already hold someone else's data, so you sometimes read a stale value or quietly corrupt unrelated data, and the real symptom shows up far away. If two pointers hold the same address and one of them frees it, the other is now dangling without ever being touched, which is one of the aliasing bugs.

4Habits that keep these out of your code

No amount of cleverness avoids these bugs. You avoid them with a few plain habits. The first three apply to almost every language. The last one is only for manual-memory code.

1. Check before you use. If a value might be null, test for it before you follow it. One check in the right place turns a crash into an ordinary branch you control. In JavaScript the optional-chaining and nullish operators do the check and the fallback in one line, which is worth learning because you will reach for it constantly:

# Python
if user is not None:
    send_email(user.address)

// JavaScript: read name only if user exists, else fall back
const name = user?.name ?? "unknown";

2. Prefer optionals over bare null. Many languages give you a type that says out loud "this might be empty", so the reader and often the type checker can see that the value could be missing and it is harder to forget. In Python you write the possibility into the signature with Optional:

# Python
from typing import Optional

def find_user(user_id: int) -> Optional[User]:
    ...   # return a User, or None if nobody matches

Other typed languages build the same idea in: Optional in Java, Option in Rust, and the ? nullable types in Kotlin and Swift, where the type checker will not let you use the value until you have handled the empty case.

3. Fail fast with a clear message. If a value should never be null at some point, check it right there and stop with a message that names what went wrong (this early check is called a guard clause), rather than letting a null travel three functions deep and blow up somewhere confusing. A crash near the cause is a gift; a crash far from the cause costs you an afternoon.

# Python
config = load_config(path)
if config is None:
    raise ValueError(f"no config found at {path}")
# ... below here you know config is real

4. Null out and null-check freed pointers (C / C++ manual-memory only). After you free a pointer, set it to null right away, so a later accidental use fails loudly as a null instead of silently reading dead memory. If you write Python, Java, Go, or C#, skip this one; the garbage collector already covers it.

Carry one working assumption through all of this: any pointer that could be empty is empty until you have checked it, and where the language offers it, let the type system carry that doubt for you.

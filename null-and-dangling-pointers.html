Null and dangling pointers

A pointer is supposed to point at something. Two ways it lets you down: it points at nothing, or it points at something that is gone. You can hold either one and pass it around safely. The trouble starts when you follow it to read what is there. Null is honest and tells you it is empty. The second kind is sneakier: it may hand you garbage without crashing at all.

The pointer page showed the basic idea: a pointer is a value that holds the address of some other data, so you can reach that data by following the pointer. This page is about the two most common ways that goes wrong. The first, null, you meet every day in every language, and it is one of the most frequent crashes in real software.

The second, dangling, you can mostly not trigger in Python or JavaScript, and it is still worth an hour of your time. It is the exact bug your garbage collector exists to prevent, so understanding it tells you what that collector is doing for you. It is behind a whole class of C and C++ crash reports you may have to read in an open-source library you depend on. And interviewers ask about it to see whether you understand what memory management is doing under you.

1Null: a pointer that points at nothing

Start with a bug you have hit. A lookup comes back empty, and the next line uses the result anyway:

# find_user can return None when nobody matches
user = find_user(user_id)
print(user.name)   # AttributeError: 'NoneType' object has no attribute 'name'

Reading user.name means "follow user to the object it points at, then read the name field." Following a pointer to use what it points at has a name: you dereference it. Here user points at nothing, so there is no object and no field to read, and Python stops.

"Points at nothing on purpose" is a value the language gives you. Maybe the data has not loaded yet, or the lookup found no match. It is called None in Python, null in Java and C, nil in Go and Ruby, nullptr in C++. Different name, same idea. Holding it is fine. Dereferencing it is the crash, and every language has its own word for the same failure:

  • Java throws a NullPointerException.
  • JavaScript says TypeError: Cannot read properties of null.
  • C or C++ usually gives a segfault, short for segmentation fault: the CPU refused an access to an address the program is not allowed to touch.

The fix is not clever. Check whether the value is there before you follow it, and decide what to do when it is not:

user = find_user(user_id)
if user is not None:
    print(user.name)
else:
    print("unknown")   # the empty case, handled on purpose

2Why null is a landmine

If the fix is one if, why is null one of the highest-frequency crashes in real software? Because the type system usually cannot tell a real value from null, and nothing forces you to check. The crash also tends to land far from the mistake. Watch a None slip through two functions before anything goes wrong:

def get_user(uid):
    return db.lookup(uid)       # returns None when nobody matches

def greeting(user):
    return "Hi " + user.name    # crashes HERE, on a None it never checked

u = get_user(999)               # u is None; no error yet
msg = greeting(u)               # the None rides in and blows up deep inside greeting

The empty value was born in get_user, but the crash happens in greeting, a function that never mentions the lookup. Now imagine that value flowing through hundreds of places in a real codebase. You have to remember the check at every one, and it is easy to miss.

Type annotations do not force the check, but they can at least warn you. A signature can lie about what it returns:

def find_user(uid) -> User:            # claims it always returns a User...
    ...                                # ...but returns None on no match. A lie.

def find_user(uid) -> Optional[User]:  # honest: a User, or None
    ...

The honest version writes the doubt into the signature. A type checker like mypy or Pyright will then flag any code that reads a field off the result without handling the None first. (Python only checks this when you actually run such a tool; the annotation alone does nothing at runtime.)

Null is not exotic. It is the ordinary case of a value that might be missing, met by code that assumed it was there. Tony Hoare, who put null into a language in 1965, later called it his "billion-dollar mistake": one small convenience that has caused countless crashes and security holes across every codebase since.

Null is your daily concern. The next pointer failure is rarer, but it is the one that explains what your garbage collector is doing and what those C crash reports are talking about.

3Dangling: a pointer to something that is gone

If you work in Python, Java, Go, JavaScript, or C#, your garbage collector will not free memory while any pointer still points at it, so this bug is mostly not yours to hit. It is a live concern in C and C++. Read this as background: it is exactly what the garbage collector is quietly saving you from.

Here is what your garbage collector spares you. In C and C++ you free memory by hand, and if you get the timing wrong you get a dangling pointer: not one pointing at nothing, the way null does, but one pointing at a real address whose data is already gone. Following it reads whatever moved in. The rest of this section shows how that happens and why it is worse than a plain crash.

The classic case is use-after-free. In C, malloc asks the system for a block of memory and hands you back its address; free gives that block back to the system to reuse. If a pointer still holds the address after the free, following it reaches memory that no longer belongs to you:

// C
int *p = malloc(sizeof(int));  // p now holds address 0x1a4c
free(p);           // block handed back; p still holds 0x1a4c
printf("%d", *p);  // use-after-free: reads whatever lives at 0x1a4c now

The same thing happens if you return the address of a local variable. Each function call gets a small temporary workspace in memory, its stack frame, and that workspace is reclaimed the moment the function returns. A pointer into it is then aimed at a slot the next call will reuse for something else.

Following a dangling pointer is worse than null in one way that is easy to miss: it might not crash. The old bytes may still be sitting there, so you read a stale value. Worse, the allocator can hand that same address to a completely different variable, so your dangling pointer now reads or overwrites data that belongs to unrelated code. A program that reads garbage without crashing can keep running, producing wrong answers for hours before anyone notices, and the real cause is nowhere near the wrong result. That silent, delayed corruption is the whole reason memory-safe languages take the choice away from you.

If two pointers hold the same address and one of them frees it, the other is now dangling without ever being touched, which is one of the aliasing bugs.

If an interviewer asksA dangling pointer holds an address whose data has already been freed. Managed languages avoid it with a garbage collector, which keeps memory alive as long as any pointer still references it and only frees it once nothing can reach it. That is the price and the point of garbage collection: you give up manual control and get safety from use-after-free.

4Habits that keep these out of your code

No amount of cleverness avoids these bugs. You avoid them with a few plain habits. The first three apply to almost every language; the last one is only for manual-memory code.

1. Check before you use

If a value might be null, test for it before you follow it. One check in the right place turns a crash into an ordinary branch you control.

# Python
if user is not None:
    send_email(user.address)

In JavaScript the same idea has cleaner syntax. ?. (optional chaining) reads user.name only if user exists, and gives undefined instead of crashing if it does not. ?? (the nullish operator) then supplies a fallback when the left side is null or undefined:

// JavaScript
const name = user?.name ?? "unknown";
// if user is null: no crash, name becomes "unknown"
2. Prefer optionals over bare null

Many typed languages give you a type that says out loud "this might be empty", so the reader and the type checker can both see the value could be missing. In Python you write it into the signature:

# Python
from typing import Optional

def find_user(uid: int) -> Optional[User]:
    ...   # a User, or None if nobody matches

Java has Optional<User>. In TypeScript you spell the possibility out in the type, and the checker will not let you read a field until you have handled the empty case:

// TypeScript
function findUser(id: number): User | null { ... }
3. Fail fast with a guard clause

If a value should never be null at some point, check it right there and stop with a message that names what went wrong. This early check is called a guard clause.

# Python
config = load_config(path)
if config is None:
    raise ValueError(f"no config found at {path}")
# ... below here you know config is real

A crash right next to the mistake tells you what to fix. A null that travels three functions deep before crashing points you at innocent code and costs you an afternoon of tracing backward. The guard trades a vague failure far away for a clear one at the cause. That trade is why this is the habit you will reach for most on your own bugs.

4. In C and C++ only: null out freed pointers

After you free a pointer, set it to null right away, so a later accidental use fails loudly as a null instead of silently reading dead memory. If you write Python, Java, Go, or C#, skip this one; the garbage collector already covers it.

Working assumptionAny pointer that could be empty is empty until you have checked it, and where the language offers it, let the type system carry that doubt for you.

5The point

Both bugs come from the same slip: you followed a pointer without asking what it was pointing at. So before you use any value another function handed back, ask one question: what does this return when there is nothing to return?

Null is your daily version of that question, answered with one check or an Optional. Dangling is the rarer, nastier version, the one where the answer is stale or corrupted memory instead of an honest empty. In a managed language your garbage collector answers it for you, which is why you almost never meet it. Knowing it exists is what lets you read a C crash report, explain what your runtime is doing, and reach for the missing case before the crash finds it for you.

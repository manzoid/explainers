How a CPU runs a program

A CPU runs a program by repeating one simple loop, very fast: fetch the next instruction, decode it (work out what it means), then execute it (carry it out). That is the whole job, the same three steps over and over.

On the last page we gathered some of the key parts: logic gates, an arithmetic unit for the math, a clock to keep everything in step, and registers. This page puts them together into something that runs a program.

1An instruction is a number

A program is a list of instructions. An instruction is a single step: one command that tells the CPU to do one specific thing. Carry out the steps in order and you have run the program.

The actions a CPU can take are simple:

  • load a number from memory into a register,
  • add two registers,
  • store a register back into memory,
  • jump to a different instruction.

Each instruction is stored in memory as a number, like the data the program works on. Part of the number says which action to take; the rest says what to act on, which registers, which memory slot.

2Memory and the program counter

To the CPU, memory is one long line of slots, each holding a number. (Physically the slots sit in a grid, but the addressing hides that, so the CPU just sees the line.) Each slot has an address: its position in the line, counting 0, 1, 2, and so on. The address is how the CPU points at a slot. It is not the number stored inside.

Each slot has an address (on top) and holds a number. The program counter points at the next instruction, here address 2.

Both the program's instructions and the data it works on live in memory. At this level they look the same, just numbers in slots. What makes a number an instruction is that the CPU reads it as one.

Memory is large, and it sits a short way from the processor. The CPU also has a few small, fast places of its own to hold numbers, called registers (we built these on the last page). There are only a handful, and the CPU keeps the numbers it is working on in them, because reaching a register is much quicker than reaching out to memory.

To run a program in order, the CPU has to track which instruction is next. It keeps that in a register of its own, the program counter, which holds the address of the next instruction.

The instructions of a program sit in consecutive addresses, in the order they should run. So "the next instruction" just means the next address: after finishing one, the CPU adds 1 to the program counter. The CPU never tells instructions apart from data; it reads whatever number is at the program counter and treats it as an instruction. The order comes from how the instructions are laid out, not from the CPU being clever about it. (Some instructions set the program counter somewhere else instead, which is how loops and branches work. More on that soon.)

3The bus

The parts have to pass numbers to each other: an instruction from memory into the CPU, a value from a register into the arithmetic unit, a result back to memory. Wiring every part straight to every other part would be a tangle.

So the parts share one set of wires, the bus. The name comes from "omnibus," Latin for "for all": every part connects to the one shared line.

Six parts on one shared bus. The program counter and instruction register are themselves registers, each set apart for its one special job; the data registers hold the numbers being worked on.

The rule is simple: at each step, one part drives the bus, puts its number on the wires, and any part that needs that number reads it off. Only one part can drive at a time, or the numbers would collide.

Who drives and who reads is decided by the control part, on each clock tick (the clock from the previous page). One tick, one transfer.

With a bus in place, moving the next instruction from memory into the CPU is just a bus transfer, which is exactly what "fetch" is.

4The loop: fetch, decode, execute

Now the parts can run a program. The CPU repeats one loop, made of three steps.

The three steps, repeated.

What each step does:

  • Fetch: bring the next instruction from memory into the CPU.
  • Decode: the control part works out which action the instruction is.
  • Execute: the control part carries that action out.

Each step is one or more transfers over the bus, and each transfer takes one tick of the clock (the clock from the last page): one part drives the bus, another reads. Here is a whole instruction, tick by tick, using "add two registers" as the example.

tickphasewhat happens on the bus
1fetchthe program counter puts the address on the bus; memory reads it
2fetchmemory puts the instruction on the bus; the instruction register reads it
3decodethe control part reads the instruction; nothing moves on the bus
4executeregister 1 puts its number on the bus; the arithmetic unit reads it
5executeregister 2 puts its number on the bus; the arithmetic unit reads it and adds
6executethe arithmetic unit puts the sum on the bus; a register reads it

One instruction (add two registers), one transfer per clock tick. Real CPUs compress and overlap these steps, but the idea holds.

Then the CPU is back at fetch, the program counter pointing at the next instruction. Fetch, decode, execute, over and over, billions of times a second. That repetition is a running program.

Copy vs share

When you write b = a, or pass a value into a function, one of two things happens. Either b gets its own separate copy, or b becomes a second name for the very same value. Which one it is decides whether changing b also changes a.

This is one of the most common sources of "wait, why did my original change?" bugs. It sits on top of two earlier pages: variables, values, and addresses (a name holds a value, or an address) and what is a pointer (an address that points at a value somewhere else). Here we use those ideas to answer one practical question: when you assign or pass something, do you copy it or share it? The bug that comes from getting this wrong has its own page, aliasing bugs.

1Two things assignment can mean

Say you have a value in a and you write b = a. There are two ways a language can carry that out.

Copy. The language makes a fresh, separate value for b that starts out equal to a. Now you have two values. Change one and the other does not move, because they are different things that just happened to be equal.

Share. The language does not make a new value. It makes b a second name for the same value a already points at. Now two names, one value. Change the value through either name and both names see the change, because there is only one thing to change.

Sharing is not a flaw; it is often what you want. Hand the same list to three parts of a program and they all work on one list, not three copies that quietly drift apart. The trouble starts only when you share without meaning to.

2Two names, one value vs two names, two values

Before any code, get the two pictures clear. Both start with the same list, [1, 2]. Then you run b.push(3). What the picture shows is where that new item lands, and whether the other name sees it.

3The usual rule: copy small, share big

Most languages do not pick one model for everything. They copy small, simple values and share large ones. A number, a single character, a true/false flag is tiny and cheap to duplicate, so the language just copies it. A list, an array, a dictionary, a bigger object can be large, so instead of duplicating the whole thing, the language shares it: it hands you another name for the same one.

So the split usually falls along the same line as the pointer page: values you hold directly get copied, values you hold through an address get shared, because copying the name copies only the address, not the thing it points at.

In most garbage-collected languages this works the same way, so you do not have to relearn it for each one. In Python, JavaScript, Java, and C#, the small scalar types (numbers, booleans, a character) copy, and everything larger (lists, arrays, objects, dictionaries) shares a reference. The names differ, Java calls the small ones primitives, but the rule does not. The exception is languages built around value types. C and C++ copy a whole struct on assignment; Go, Swift, and Rust copy their struct and array types by default too. Go slices are the usual trap: a slice copies a small header but shares the underlying array, so it acts shared right up until an append forces a reallocation. If you work in one of these languages, check the default rather than assume sharing.

In a garbage-collected language, assume plain numbers are copied and lists, arrays, and objects are shared. In C, C++, Go, Swift, or Rust, check first: these often copy the whole thing.

4The classic surprise: a function changes your list

Passing an argument follows the same rule as assignment. When you pass a shared value into a function, the parameter inside the function is just another name for your value. If the function changes it in place, your original changed too. Here it is in Python:

def add_item(items):
    items.append("new")     # changes the one list

cart = ["a", "b"]
add_item(cart)
print(cart)                 # ['a', 'b', 'new']  <-- yours changed

The function never returned anything and you never reassigned cart, yet cart grew. That is because items and cart were two names for one list the whole time, and append changed that one list. JavaScript does the same when you pass an array to a function that pushes onto it:

function addItem(items) {
  items.push("new");        // changes the one array
}

const cart = ["a", "b"];
addItem(cart);
console.log(cart);          // ["a", "b", "new"]  <-- yours changed

Note the difference between changing the value in place and rebinding the name. items.append(...) changes the shared list, so the caller sees it. Writing items = ["something", "else"] only points the local name items at a new list; the caller's cart is untouched. Changing the thing is shared; repointing the name is local.

5Getting a copy on purpose

When you want your own separate thing, you have to ask for a copy; the language will not do it for you. In JavaScript the two you will reach for are:

const b = [...arr];          // spread into a new array
const b = Array.from(arr);   // same idea

Python has its own: list(x), x[:], or x.copy(). Other languages have similar.

One catch applies to all of them. These make a shallow copy: they duplicate the outer container but leave the inner objects shared. If your list holds other lists, the outer list is now separate, but the inner lists are still the same ones. So this still surprises people:

inner = [1, 2]
a = [inner]
b = a.copy()          # shallow: new outer list, same inner list
b[0].append(3)
print(a)              # [[1, 2, 3]]  <-- a changed too

For a copy that stays separate all the way down, you need a deep copy:

b = copy.deepcopy(a)          # Python (import copy)
const b = structuredClone(a); // modern JavaScript, ES2022 and up

Java has no single built-in for this; people use a copy constructor, copy each field by hand, or reach for a library. Which one you need depends on how deep your changes reach: a shallow copy is enough if you only touch the outer container, a deep copy is needed once you change something nested.

6Checking whether two names share

When you are not sure whether two names point at one object or two, ask the language directly instead of guessing. In Python, a is b is true only when they are the very same object, sharing one address in memory. In JavaScript, a === b tells you the same for objects: two arrays, or two plain objects, are === only when they are the one object, not merely equal-looking. In Java, a == b on objects is that same identity check, while a.equals(b) is the value check; mixing up the two is the most common form of exactly this confusion.

One caution, since the last sections said numbers get copied: these identity rules are about objects. On plain numbers and other primitives, === and == compare the value, so 5 === 5 is true because the values match, not because there is one shared object behind them. Identity only becomes a real question once you hold something through an address.

If an identity check comes back true, a change through one name will show under the other, and that is your warning to make a copy if you did not want that. The bug this whole question produces, and how to spot and avoid it, is the subject of the aliasing bugs page. The values most often caught by it, lists and arrays, get their own treatment on the arrays page.

Copy vs share

You write b = a, or you pass a value into a function. One of two things happens. Either b gets its own separate copy, or b becomes a second name for the very same value. Which one it is decides whether changing b also changes a.

The setup in one sentence: a number lives right inside your variable, but a list or an object lives elsewhere and your variable holds its address, and that address is the kind of value that gets shared. (For the mechanics of that, see variables, values, and addresses.)

1Two things assignment can mean

Say you have a value in a and you write b = a. There are two ways a language can carry that out.

Copy. The language makes a fresh, separate value for b that starts out equal to a. Now you have two values. Change one and the other does not move, because they are different things that just happened to be equal.

Share. The language does not make a new value. It makes b a second name for the value a already points at. Now two names, one value. Change the value through either name and both names see the change, because there is only one thing to change.

Sharing is not a flaw; it is often what you want. Hand the same list to three parts of a program and they all work on one list, not three copies that quietly drift apart. The trouble starts only when you share without meaning to.

2What sharing and copying look like

Both cases start the same way: a and b name one list, [1, 2]. Then you run b.append(3).

b = a
b.append(3)     # does a change too?

Read the picture in two halves. On the share side, watch how both names still point at one list, so the new item shows up under both. On the copy side, watch b sit on its own separate list, so only b grows.

So the whole question is which side you are on. If b = a shared, that one append reached a. If it copied, a never moved. The next section is the rule that tells you which one your language did.

3The usual rule: copy small, share big

In a garbage-collected language, assume plain numbers are copied and lists, arrays, and objects are shared.

Most languages do not pick one model for everything. They copy small, simple values and share large ones. A number, a single character, a true/false flag is tiny and cheap to duplicate, so the language just copies it. A list, an array, a dictionary, a bigger object can be large, so instead of duplicating the whole thing, the language shares it: it hands you another name for the same one.

This follows the split from the setup: values you hold directly get copied, values you hold through an address get shared, because copying the name copies only the address, not the thing it points at. Python and JavaScript both work this way, and so do Java and C#. (Some statically typed languages copy the whole struct or array by default instead; check the docs if you work in one.)

When you are not sure which one you got, do not guess. Section 6 shows how to ask the language directly.

4The classic surprise: a function changes your list

Passing an argument follows the same rule as assignment. When you pass a shared value into a function, the parameter inside the function is just another name for your value. If the function changes it in place, your original changed too. Here it is in Python:

def add_item(items):
    items.append("new")     # changes the one list

cart = ["a", "b"]
add_item(cart)
print(cart)                 # ['a', 'b', 'new']  <-- yours changed

The function never returned anything and you never reassigned cart, yet cart grew. That is because items and cart were two names for one list the whole time, and append changed that one list. JavaScript behaves identically: pass an array to a function that calls items.push("new"), and the caller's array grows the same way.

If you suspect this is happening, confirm it before you change anything. An identity check (Python's is, shown in section 6) comes back true here, telling you the parameter and your variable are one object, so any change will reach the caller. This is a common interview probe: an interviewer may show a helper that quietly edited the caller's list and ask why, or ask how you would stop it.

Changing a value in place is shared; rebinding a name is local. items.append(x) changes the one shared list, so the caller sees it. items = [...] only points the local name items at a new list, and the caller's variable is untouched.

5Getting a copy on purpose

When you want your own separate thing, you have to ask for a copy; the language will not do it for you. The idioms are short in both languages:

Python:

b = list(x)     # new list from x
b = x[:]        # same idea, slice the whole thing
b = x.copy()    # same idea, spelled out

JavaScript:

const b = [...arr];        // spread into a new array
const b = Array.from(arr); // same idea

One catch applies to all of them. These make a shallow copy: they duplicate the outer container but leave the inner objects shared. If your list holds other lists, the outer list is now separate, but the inner lists are still the same ones. So this still surprises people:

inner = [1, 2]
a = [inner]
b = a.copy()          # shallow: new outer list, same inner list
b[0].append(3)
print(a)              # [[1, 2, 3]]  <-- a changed too

For a copy that stays separate all the way down, you need a deep copy:

Python:

import copy
b = copy.deepcopy(a)

JavaScript:

const b = structuredClone(a);   // ES2022 and up

Which one you need depends on how deep your changes reach: a shallow copy is enough if you only touch the outer container, a deep copy is needed once you change something nested.

6Checking whether two names share

When you are not sure whether two names point at one object or two, ask the language instead of guessing. In Python, write a is b: it is true only when the two names are the very same object, sharing one address. In JavaScript, a === b does the same for objects: two arrays, or two plain objects, are === only when they are the one object, not merely equal-looking. (Java splits these apart: a == b on objects is the identity check, and a.equals(b) is the value check. Mixing up the two is this exact confusion.)

If the check comes back true, a change through one name will show under the other. That is your signal to make a copy if you did not want that.

One footnote: these checks are about objects. On plain numbers and other small values, is, ===, and == compare the value itself, so 5 == 5 is true because the values match, not because one object is shared. Identity only becomes a real question once you hold something through an address.

The bug this whole question tends to produce, and how to spot and avoid it, has its own page: aliasing bugs.

Copy when you need your own separate thing. Let sharing stand when several parts should work on one value. And when something changes that you did not touch, check identity first.
Reuse and share freely.

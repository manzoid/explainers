Copy vs share

When you write b = a, or pass a value into a function, one of two things happens. Either b gets its own separate copy, or b becomes a second name for the very same value. Which one it is decides whether changing b also changes a.

This is one of the most common sources of "wait, why did my original change?" bugs. It sits on top of two earlier pages: variables, values, and addresses (a name holds a value, or an address) and what is a pointer (an address that points at a value somewhere else). Here we use those ideas to answer one practical question: when you assign or pass something, do you copy it or share it? The bug that comes from getting this wrong has its own page, aliasing bugs.

1Two things assignment can mean

Say you have a value in a and you write b = a. There are two ways a language can carry that out.

Copy. The language makes a fresh, separate value for b that starts out equal to a. Now you have two values. Change one and the other does not move, because they are different things that just happened to be equal.

Share. The language does not make a new value. It makes b a second name for the same value a already points at. Now two names, one value. Change the value through either name and both names see the change, because there is only one thing to change.

Sharing is not a flaw; it is often what you want. Hand the same list to three parts of a program and they all work on one list, not three copies that quietly drift apart. The trouble starts only when you share without meaning to.

2The picture

3The usual rule: copy small, share big

Most languages do not pick one model for everything. They copy small, simple values and share large ones. A number, a single character, a true/false flag is tiny and cheap to duplicate, so the language just copies it. A list, an array, a dictionary, a bigger object can be large, so instead of duplicating the whole thing, the language shares it: it hands you another name for the same one.

So the split usually falls along the same line as the pointer page: values you hold directly get copied, values you hold through an address get shared, because copying the name copies only the address, not the thing it points at.

4The rule holds across languages

You do not have to memorize this per language, because it is nearly the same everywhere. The same pattern holds in Python, JavaScript, Java, C#, and Go: built-in scalar types (numbers, booleans, a character) copy, and everything larger (lists, arrays, objects, dictionaries) shares a reference. The names differ, Java calls the small ones primitives, but the rule does not.

When in doubt, assume plain numbers and the like are copied, and lists, arrays, and objects are shared.

5The classic surprise: a function changes your list

Passing an argument follows the same rule as assignment. When you pass a shared value into a function, the parameter inside the function is just another name for your value. If the function changes it in place, your original changed too.

function addItem(items) {
  items.push("new");        // changes the one array
}

const cart = ["a", "b"];
addItem(cart);
console.log(cart);          // ["a", "b", "new"]  <-- yours changed

The function never returned anything and you never reassigned cart, yet cart grew. That is because items and cart were two names for one array the whole time, and push changed that one array. The same thing happens in Java when you pass a list to a method that appends to it:

void addItem(List<String> items) {
    items.add("new");       // changes the one list
}

List<String> cart = new ArrayList<>(List.of("a", "b"));
addItem(cart);
System.out.println(cart);   // [a, b, new]  <-- yours changed

Note the difference between changing the value in place and rebinding the name. items.push(...) changes the shared array, so the caller sees it. Writing items = ["something", "else"] only points the local name items at a new array; the caller's cart is untouched. Changing the thing is shared; repointing the name is local.

6Getting a copy on purpose

When you want your own separate thing, you have to ask for a copy; the language will not do it for you. In JavaScript the two you will reach for are:

const b = [...arr];          // spread into a new array
const b = Array.from(arr);   // same idea

Python has its own: list(x), x[:], or x.copy(). Other languages have similar. One catch applies to all of them: these make a shallow copy, a new outer container whose slots still point at the same inner objects. If your list holds other lists, the outer one is now separate but the inner ones are still shared. For a fully independent copy all the way down, you need a deep copy (in Python, copy.deepcopy; in JavaScript, structuredClone). Which one you need depends on how deep the changes reach.

7Checking whether two names share

When you are not sure whether two names point at one object or two, ask the language directly instead of guessing. In Python, a is b is true only when they are the very same object. In JavaScript, a === b on two objects is true only when they are the same object, not merely equal-looking. If that check comes back true, a change through one name will show under the other, and that is your warning to make a copy if you did not want that.

The bug this whole question produces, and how to spot and avoid it, is the subject of the aliasing bugs page. The values most often caught by it, lists and arrays, get their own treatment on the arrays page.
